Q1：为什么要把 PDF 解析拆成两个类？

PDFParser 负责"粗解析"，JsonReportProcessor 负责"细组装"。这是一种什么设计模式？如果不拆分会有什么问题？

💡 提示：想想下游模块需要什么格式的数据，以及"解析"和"格式化"能不能混在一起。
📝 点击查看答案
两阶段架构是"关注点分离"设计——把"取数据"和"格式化数据"解耦：

阶段1 — PDFParser.convert_documents()：取原始数据
def convert_documents(self, input_doc_paths):
    # 一行代码：把 PDF 文件列表喂给 Docling，拿到原始解析结果
    conv_results = self.doc_converter.convert_all(source='input_doc_paths')
    return conv_results  # Iterable[ConversionResult]，每个 PDF 一个
这个阶段完全依赖 Docling API（pdf_backend、PipelineOptions、TableFormer），返回的是 Docling 内部格式，下游根本看不懂。

阶段2 — JsonReportProcessor.assemble_report()：格式化为下游JSON
def assemble_report(self, conv_result, normalized_data=None):
    data = normalized_data if normalized_data else conv_result.document.export_to_dict()
    assembled_report = {}
    assembled_report['metainfo'] = self.assemble_metainfo(data)     # → 页数/公司名
    assembled_report['content']  = self.assemble_content(data)       # → 逐页文本
    assembled_report['tables']   = self.assemble_tables(tables,data) # → MD+HTML+JSON
    assembled_report['pictures'] = self.assemble_pictures(data)       # → 图片位置
    return assembled_report
这个阶段与 PDF 引擎零依赖——它只接收 dict，返回 dict。

不拆分会怎样？——三个具体问题：
① 换引擎代价巨大：Docling→PyMuPDF 需要重写所有代码，包括 JSON 格式化逻辑
② 单元测试困难：测试"公司名是否正确放入 metainfo"需要真正跑一次 PDF 解析，一个测试几十秒
③ 并行模式冲突：串行模式中 convert_documents() 在主进程执行，process_documents() 也在主进程；但并行模式中，_process_chunk() 在子进程中同时调用两者。如果它们耦合在一起，子进程调用逻辑会极其混乱

这正是 parse_and_export() 中分两步的原因： 
def parse_and_export(self, input_doc_paths):
    conv_results = self.convert_documents(input_doc_paths)      # 阶段1：全部转换
    success, failure = self.process_documents(conv_results)     # 阶段2：逐个组装+存盘
    return success, failure

Q2：expand_groups() 为什么要"拍平"嵌套列表？

Docling 会把页面内容按层级组织（list → group → label/group → ...），不拍平会怎样？能举一个"嵌套结构"的具体例子吗？

💡 提示：想想看如果一篇文章有"大标题 + 小标题 + 正文"三层结构，拍平前后分别是什么样子。
📝 点击查看答案
Docling 保留 PDF 的 visual hierarchy，expand_groups() 将它"压扁"成线性列表。先看核心代码：

def expand_groups(self, body_children, groups):
    expanded_children = []
    for item in body_children:
        if isinstance(item, dict) and '$ref' in item:
            ref = item['$ref']                          # 例如 "#/groups/0"
            ref_type, ref_num = ref.split('/')[-2:]      # → ('groups', '0')
            ref_num = int(ref_num)

            if ref_type == 'groups':                     # 这是一个 group 引用！
                group = groups[ref_num]                  # 取出 group 对象
                group_id = ref_num
                group_name = group.get('name', '')       # 如 "主营业务"
                group_label = group.get('label', '')     # 如 "section"

                for child in group['children']:          # 遍历 group 内的子元素
                    child_copy = child.copy()            # 浅拷贝避免污染
                    child_copy['group_id'] = group_id    # 附加来源信息
                    child_copy['group_name'] = group_name
                    child_copy['group_label'] = group_label
                    expanded_children.append(child_copy) # 展开到顶层
            else:
                expanded_children.append(item)           # 非 groups 直接保留
        else:
            expanded_children.append(item)
    return expanded_children
具体例子——年报中的层级列表：

年报第3页有这样一段内容：
1. 主营业务数据
1.1 营收构成
产品A收入60亿元，占总营收60%
1.2 成本构成
原材料成本30亿元
2. 研发投入
研发费用15亿元，同比增长20%
Docling 内部把这段内容组织为： 
data['body']['children'] = [
    {"$ref": "#/groups/0"},          # ← 整个"1.主营业务数据" group
    {"$ref": "#/groups/1"},          # ← 整个"2.研发投入" group
]
data['groups'] = [
    {  # groups[0] — "1.主营业务数据" 及其子项
        "name": "1. 主营业务数据", "label": "section",
        "children": [
            {"$ref": "#/groups/2"},      # ← 嵌套的 group "1.1营收构成"
            {"$ref": "#/groups/3"},      # ← 嵌套的 group "1.2成本构成"
        ]
    },
    {  # groups[1] — "2.研发投入"
        "name": "2. 研发投入", "label": "section",
        "children": [{"$ref": "#/texts/5"}]  # → 研发费用文本
    },
    {  # groups[2] — "1.1营收构成"
        "name": "1.1 营收构成", "label": "subsection",
        "children": [{"$ref": "#/texts/3"}]  # → 产品A收入文本
    },
    {  # groups[3] — "1.2成本构成"
        "name": "1.2 成本构成", "label": "subsection",
        "children": [{"$ref": "#/texts/4"}]  # → 原材料成本文本
    }
]
expand_groups() 处理后： 
expanded_children = [
    # groups[0] 展开 → 遍历其 children → 遇到 groups[2], groups[3]
    {"$ref": "#/groups/2", "group_id":0, "group_name":"1. 主营业务数据", "group_label":"section"},
    {"$ref": "#/groups/3", "group_id":0, "group_name":"1. 主营业务数据", "group_label":"section"},
    # groups[1] 直接展开 → texts[5] 被提升
    {"$ref": "#/texts/5",  "group_id":1, "group_name":"2. 研发投入", "group_label":"section"},
    # 注意！groups[2] 和 groups[3] 也会被递归处理（它们出现在 expanded_children 中）：
    # groups[2] → text[3] 提升：
    {"$ref": "#/texts/3",  "group_id":2, "group_name":"1.1 营收构成", "group_label":"subsection"},
    # groups[3] → text[4] 提升：
    {"$ref": "#/texts/4",  "group_id":2, "group_name":"1.2 成本构成", "group_label":"subsection"},
]
等等——代码只遍历了 body.children，没有递归处理 group 内部的 group！这是关键设计：
展开逻辑只对 body.children 做一层展开。如果 group 内部再嵌套 group（如 groups[0] → groups[2]），内部的 group 引用会被保留在 expanded_children 中。
但在 assemble_content() 的第二步遍历中，会再次遇到 {"$ref": "#/groups/2"}，此时它不是从 body.children 来，而是从 expanded_children 来——但 assemble_content() 的第二步遍历只会匹配 ref_type == 'texts'/'tables'/'pictures'，不会再递归处理 groups。

不拍平的后果：下游 text_splitter 收到 {"$ref": "#/groups/0"} 这种东西，根本无法将其转为文本 chunk。embedding 也会把 label 和 text 混在一起编码，导致语义失真。

Q3：$ref 引用是什么意思？为什么 Docling 要这样设计？

为什么 Docling 不直接把文本内容嵌入在元素里，而是用一个引用编号？

💡 提示：想想同一段文本出现在多个位置（比如页眉页脚重复出现）的情况。_process_text_reference()
📝 点击查看答案
$ref = JSON Reference，是 Docling 数据库范式化设计的核心。先看解析它的代码：

# assemble_content() 中解析 $ref 的关键代码
for item in expanded_children:
    if isinstance(item, dict) and '$ref' in item:
        ref = item['$ref']                     # 例如 "#/texts/5"
        ref_type, ref_num = ref.split('/')[-2:] # → ('texts', '5')
        ref_num = int(ref_num)                  # → 5

        if ref_type == 'texts':
            text_item = data['texts'][ref_num]  # 按编号取到真实文本
            content_item = self._process_text_reference(ref_num, data)
            # _process_text_reference 内部：
            #   text_item = data['texts'][ref_num]
            #   content_item = {'text': text_item['text'], 'type': text_item['label'], 'text_id': ref_num}
            #   如果 text != orig，附加 orig 字段（OCR纠正前的内容）
            #   如果存在 enumerated/marker，附加列表编号/标记
为什么不用"内嵌"而用"引用"？——4个具体原因：

① 内容去重——页眉页脚
一份年报100页，每页都出现页眉 "XX股份有限公司 2024年年度报告"。内嵌式需复制100次；$ref 只存1次： 
data['texts'][0] = {"text": "XX股份有限公司 2024年年度报告", "label":"header"}
// body.children 中每页引用一次：
[{"$ref":"#/texts/0"}, ..., {"$ref":"#/texts/0"}]  ← 100次引用，0次重复存储
② 内容与结构分离
body.children 只存阅读顺序（哪个在前哪个在后），texts[] 只存储内容本身。改变顺序不需要改内容，修正错别字不需要动顺序。

③ 跨页文本
一段话"公司2024年度实现了良好的经营业绩，其中主营业务收入100亿元..."跨第1-2页。Docling 识别为同一文本对象，prov 记录两个位置： 
texts[5] = {"text":"公司2024年度实现...100亿元","prov":[
    {"page_no":1,"bbox":{...}},  # 前半段在第1页
    {"page_no":2,"bbox":{...}}   # 后半段在第2页
]}
assemble_content() 只取 prov[0].page_no（起始页），所以跨页文本归到第1页。

④ 可复用的解析器
_process_text_reference() 被两处调用：assemble_content() 处理正文文本，_process_picture_block() 处理图片附属文本（图注）。如果文本内嵌，两处各写一套解析逻辑，容易不一致。 类比数据库：body.children = 订单明细表（存外键），texts[] = 商品表（存唯一内容），$ref = 外键 FOREIGN KEY (text_id) REFERENCES texts(id)。宁可多一次 JOIN，也不要在N个地方存同一份数据。

Q4：表格为什么输出 Markdown / HTML / JSON 三种格式？

assemble_tables() 同时生成三种格式。下游 RAG 每一步分别需要哪种格式？为什么不能只保留一种？

💡 提示：回顾一下 text_splitter 用的是什么格式，retrieval 用的是什么，最终给 LLM 看的又是什么。
📝 点击查看答案
assemble_tables() 同时输出三种格式，因为 RAG 管线的三个环节各需一种：

def assemble_tables(self, tables, data):
    assembled_tables = []
    for i, table in enumerate(tables):
        table_json_obj = table.model_dump()   # ① JSON——完整原始数据
        table_md = self._table_to_md(table_json_obj)  # ② Markdown——文本化
        table_html = table.export_to_html()            # ③ HTML——保留结构

        table_page_num = data['tables'][i]['prov'][0]['page_no']
        table_obj = {
            'table_id': int(data['tables'][i]['self_ref'].split('/')[-1]),
            'page': table_page_num,
            'bbox': [...],
            '#-rows': data['tables'][i]['data']['num_rows'],
            '#-cols': data['tables'][i]['data']['num_cols'],
            'markdown': table_md,     # ← 消费方：parsed_reports_merging.py
            'html': table_html,       # ← 消费方：tables_serialization.py → GPT-4o-mini
            'json': table_json_obj    # ← 消费方：调试/程序化提取
        }
        assembled_tables.append(table_obj)
    return assembled_tables
三个消费路径，三种格式，缺一不可：

路径A — Markdown → 页文本 → text_splitter → FAISS
parsed_reports_merging.py 的 _get_serialized_table_text() 读取 table["markdown"] 并拼入页文本： 
# 输出示例：嵌入在 page.text 中的 Markdown 表格
| 项目    | 2024年 | 2023年 | 变化 |
|---------|--------|--------|------|
| 营业收入 | 100亿  | 80亿   | +25% |
| 净利润   | 15亿   | 12亿   | +25% |
为什么是 Markdown？embedding 模型对 Markdown 表格的语义理解最好——"营业收入"和"100亿"在同一行中，向量能编码这种关联。HTML 标签（<td>）是噪声，JSON 有大量元数据字段也是噪声。

路径B — HTML → GPT-4o-mini → information_blocks → FAISS
tables_serialization.py 中 table_content = table_info["html"]，直接发给 LLM： 
query = f'Here is a table in HTML format:\n"""{table_content}"""'
为什么是 HTML？Markdown 不支持 colspan="3"（合并单元格），年报财务报表中"营业总成本"横跨3列，Markdown 会把它"摊平"成3个重复的"营业总成本"，LLM 会误解为3个独立项目。

路径C — JSON → 调试 / 数值计算
table_json_obj = {
  "data": {"grid": [[{"text":"营业收入"},{"text":"100亿"}], [{"text":"净利润"},{"text":"15亿"}]]],
  "num_rows": 3, "num_cols": 2
}
如果 LLM 回答不对，用 JSON 对比原始 grid 数据定位是"Docling 解析错了"还是"Markdown 格式化错了"。如需做精确数值计算（所有公司营收加总），遍历 JSON grid 比解析 Markdown 字符串可靠。 只用一种的后果：
只 Markdown → 合并单元格信息丢失，LLM 看不懂复杂表格
只 HTML → text_splitter 散标签，embedding 质量骤降
只 JSON → LLM 无法直接理解网格语义，需要额外解释层

Q5：_process_chunk() 为什么必须是模块级函数？

文档提到了 pickle 序列化限制。主进程怎么把任务"分发"给子进程？子进程之间共享数据吗？

💡 提示：如果写成类方法，主进程 pickle 时会报错 "cannot pickle 'method' object"。ProcessPoolExecutor
📝 点击查看答案
Python 多进程的核心限制：类实例方法不能被 pickle 序列化。先看代码结构：

# ✅ 正确的写法——模块级函数
def _process_chunk(pdf_paths, pdf_backend, output_dir, num_threads,
                    metadata_lookup, debug_data_path):
    parser = PDFParser(pdf_backend=pdf_backend, output_dir=output_dir, ...)
    parser.metadata_lookup = metadata_lookup
    parser.debug_data_path = debug_data_path
    parser.parse_and_export(pdf_paths)
    return f"Processed {len(pdf_paths)} PDFs."

# ❌ 如果写成类方法——会报错
class PDFParser:
    def _process_chunk(self, pdf_paths):  # self 无法 pickle！
        self.parse_and_export(pdf_paths)
为什么类方法不能被 pickle？pickle.dumps(self._process_chunk) 时，Python 需要序列化 self，而 self 包含了整个 PDFParser 实例的所有属性（doc_converter、metadata_lookup、output_dir...），其中 doc_converter 内部有 C++ 扩展对象（Docling 的底层引擎），这些对象不支持 pickle。

分发流程——看 parse_and_export_parallel() 的完整逻辑： 
def parse_and_export_parallel(self, input_doc_paths, optimal_workers=10, chunk_size=None):
    cpu_count = multiprocessing.cpu_count()
    if optimal_workers is None:
        optimal_workers = min(cpu_count, len(input_doc_paths))
    if chunk_size is None:
        chunk_size = max(1, len(input_doc_paths) // optimal_workers)

    # 步骤1：切分 PDF 列表
    chunks = [input_doc_paths[i:i+chunk_size] for i in range(0, total_pdfs, chunk_size)]

    # 步骤2：提交到进程池
    with ProcessPoolExecutor(max_workers=optimal_workers) as executor:
        futures = [
            executor.submit(
                _process_chunk,       # ← 只传函数引用（不含 self）
                chunk,                # ← List[Path]，可pickle
                self.pdf_backend,     # ← 类对象（DoclingParseV2DocumentBackend），可pickle
                self.output_dir,      # ← Path 对象，可pickle
                self.num_threads,     # ← int，可pickle
                self.metadata_lookup, # ← dict，可pickle（但可能很大）
                self.debug_data_path  # ← Path，可pickle
            )
            for chunk in chunks
        ]

        # 步骤3：as_completed 实时收集
        for future in as_completed(futures):
            result = future.result()   # "Processed 5 PDFs."
            processed_count += int(result.split()[1])
数据不共享——每个子进程独立内存空间：
① 主进程 pickle 所有参数 → 写入管道 → 子进程从管道读取 unpickle
② 子进程 独立创建 PDFParser() 实例（加载 Docling 模型到自己的内存）
③ metadata_lookup 被复制（不是共享），子进程修改它不影响主进程
④ 返回结果通过 return + pickle 传回主进程

关键设计决策：参数中传 self.pdf_backend（类）而不是 self（实例），因为类可以被 pickle，含状态的实例不能。

Q6：并行模式和串行模式的区别是什么？什么时候选哪个？

并行vs串行的本质区别：任务调度模型不同。先看两个模式的代码对比：

def parse_and_export_parallel(self, input_doc_paths, optimal_workers=10, chunk_size=None):

    chunks = [input_doc_paths[i:i+chunk_size] for i in range(0, total_pdfs, chunk_size)]

    with ProcessPoolExecutor(max_workers=optimal_workers) as executor:
        futures = [executor.submit(_process_chunk, chunk, ...) for chunk in chunks]
        for future in as_completed(futures):
            result = future.result()

PDF2 解析: 6s
JSON组装+写盘: ≈0.5s × 2

_normalize_page_sequence() 解决"页码空洞"问题，确保下游按 page 索引不出错。完整代码：

def _normalize_page_sequence(self, data: dict) -> dict:
    if 'content' not in data:
        return data              # 防御：无 content 字段直接返回

    normalized_data = data.copy()
    existing_pages = {page['page'] for page in data['content']}  # 所有已有页码
    max_page = max(existing_pages)                                # 最大页码

    empty_page_template = {"content": [], "page_dimensions": {}}  # 空页模板

    new_content = []
    for page_num in range(1, max_page + 1):          # 从1遍历到max_page
        page_content = next(
            (page for page in data['content'] if page['page'] == page_num),
            {"page": page_num, **empty_page_template}  # 找不到 → 用空模板
        )
        new_content.append(page_content)

    normalized_data['content'] = new_content
    return normalized_data

② 纯图片页：架构图、流程图等，Docling 可能只提取图片不提取文字

def _process_text_reference(self, ref_num, data):
    text_item = data['texts'][ref_num]            # 取出原始文本项
    content_item = {
        'text': text_item.get('text', ''),        # 文本内容
        'type': text_item['label'],                # paragraph/heading/list_item/...
        'text_id': ref_num                         # 原始编号（用于溯源）
    }

    # OCR 纠正前后文本不同 → 保留原始版本
    if text_item.get('orig', '') != text_item.get('text', ''):
        content_item['orig'] = text_item['orig']

    if 'enumerated' in text_item:
        content_item['enumerated'] = text_item['enumerated']  # 是否有序列表
    if 'marker' in text_item:
        content_item['marker'] = text_item['marker']          # 列表标记 "•" / "1."

    return content_item

class PyMuPDFAdapter:
    """将 PyMuPDF 的页面文本转为 Docling 兼容的 data dict"""
    def convert(self, pymupdf_pages) -> dict:
        data = {"body": {"children": []}, "groups": [], "texts": [], "tables": [], "pictures": []}
        for page_num, page in enumerate(pymupdf_pages):
            for block in page["blocks"]:
                if block["type"] == 0:  # 文本块
                    text_id = len(data["texts"])
                    data["texts"].append({
                        "text": " ".join([span["text"] for span in block["lines"][0]["spans"]]),
                        "label": "paragraph",
                        "prov": [{"page_no": page_num+1, "bbox": block["bbox"]}]
                    })
                    data["body"]["children"].append({"$ref": f"#/texts/{text_id}"})
        return data

① _process_chunk() — 模块级函数 PARALLEL HELPER

🤔 这个函数做什么？（大白话）

当启用并行模式时，主进程会把 PDF 文件分成好几堆（chunk）， 然后每堆交给一个子进程去处理。 这个函数就是子进程的"工作任务"——它在子进程里创建一个新的 PDFParser， 调用串行解析方法处理分配给它的那堆 PDF。

💡 为什么必须是模块级函数（不能写成类方法）？
  • Python 的 ProcessPoolExecutor 需要把任务函数 pickle（序列化） 后传给子进程
  • 类成员方法无法被 pickle，所以必须写成模块级的独立函数
  • 每个子进程有自己独立的 Python 解释器，所以可以各自创建自己的 PDFParser 实例
🐍 关键代码 
def _process_chunk(pdf_paths, pdf_backend, output_dir, num_threads, metadata_lookup, debug_data_path):
    """辅助函数：在独立的子进程中处理一批 PDF 文件"""
    parser = PDFParser(pdf_backend=pdf_backend, output_dir=output_dir,
                       num_threads=num_threads, csv_metadata_path=None)
    parser.metadata_lookup = metadata_lookup  # 从主进程"继承"元数据
    parser.debug_data_path = debug_data_path
    parser.parse_and_export(pdf_paths)        # 调用串行方法处理这批 PDF
    return f"Processed {len(pdf_paths)} PDFs."
📍 它在流程中的位置： 只在 parse_and_export_parallel() 中被调用 → 作为 executor.submit() 的任务函数

③ _parse_csv_metadata() — 静态方法 CSV

🤔 这个函数做什么？（大白话）

读一个 CSV 文件，里面有两列：sha1（PDF 的文件名哈希）和 company_name（公司名）。 把每一行转成 {sha1: {company_name: "某公司"}} 的字典， 这样后面处理 PDF 时就能根据文件名知道是哪个公司的年报。

🐍 关键代码 
@staticmethod
def _parse_csv_metadata(csv_path: Path) -> dict:
    import csv
    metadata_lookup = {}
    with open(csv_path, 'r', encoding='utf-8') as csvfile:
        reader = csv.DictReader(csvfile)  # 每行自动转为 {列名: 值}
        for row in reader:
            company_name = row.get('company_name', row.get('name', '')).strip('"')
            metadata_lookup[row['sha1']] = {'company_name': company_name}
    return metadata_lookup  # {"abc123...": {"company_name": "阿里巴巴"}}

⑨ parse_and_export_parallel() — 并行解析入口 PARALLEL ENTRY

🤔 这个函数做什么？（大白话）

这是并行模式的入口——当 PDF 很多时（比如100份年报），串行太慢。 它把 PDF 列表切分成几个 chunk，用 ProcessPoolExecutor 启动多个独立子进程同时解析。每个子进程运行 _process_chunk()。

🐍 关键代码 — 三步并行策略 
# 步骤1：计算最优参数
cpu_count = multiprocessing.cpu_count()
optimal_workers = min(cpu_count, total_pdfs)          # 进程数 ≤ CPU核心数
chunk_size = max(1, total_pdfs // optimal_workers)     # 每个进程处理几个PDF

# 步骤2：切分PDF列表
chunks = [input_doc_paths[i:i+chunk_size] for i in range(0, total_pdfs, chunk_size)]

# 步骤3：提交到进程池并行执行
with ProcessPoolExecutor(max_workers=optimal_workers) as executor:
    futures = [executor.submit(_process_chunk, chunk, ...) for chunk in chunks]
    for future in as_completed(futures):
        result = future.result()  # 哪个chunk先完成就先处理哪个
📖 并行策略详解
  • optimal_workers = min(CPU核心数, PDF总数)：16核机器处理100个PDF → 16个进程，每个处理约6-7个
  • 为什么用多进程而不是多线程？PDF解析是CPU密集型，Python的GIL锁会让多线程等于单线程。多进程每个进程有独立Python解释器，真正并行
  • as_completed 实时追踪：不按提交顺序等待，哪个进程先跑完就先报告进度
📍 它在流程中的位置： 用户直接调用的并行入口。内部通过 ProcessPoolExecutor → _process_chunk → parse_and_export 完成解析。

⑮ expand_groups() — 展开分组结构 UTILITY

🤔 这个函数做什么？（大白话）

Docling 会把 PDF 中的列表项（如项目符号列表）放在 groups 数组中。 这个函数遍历 body.children，遇到 groups 类型的引用时， 把 group 里的每个子元素"拆出来"，"拍平"到顶层。 这样后续处理就不用管嵌套了，每个内容项都是平等的。

⑯ _process_text_reference() — 文本引用处理 UTILITY

🤔 这个函数做什么？（大白话）

根据引用编号（如 #/texts/5 中的 5），从 data['texts'] 数组中取出第5个文本块， 然后构建一个标准化的内容字典：包含 text（文本内容）、type（是paragraph还是heading）、 text_id（编号，用于溯源）。如果 OCR 纠正了原文，还附带 orig 字段。

🐍 关键代码 
def _process_text_reference(self, ref_num, data):
    text_item = data['texts'][ref_num]               # 根据编号取文本块
    content_item = {
        'text': text_item.get('text', ''),            # 文本内容
        'type': text_item['label'],                    # 类型：paragraph/heading/footnote
        'text_id': ref_num                             # 编号（用于溯源）
    }
    # 如果OCR纠正了原文（orig≠text），附上原始内容
    if text_item.get('orig', '') != text_item.get('text', ''):
        content_item['orig'] = text_item['orig']
    # 附加上下文标记（列表编号、标记符号等）
    if 'enumerated' in text_item:
        content_item['enumerated'] = text_item['enumerated']   # 是否有序列表
    if 'marker' in text_item:
        content_item['marker'] = text_item['marker']           # 如 '•', '1.'
    return content_item

        if ref_type == 'texts':                       # 处理文本
            content_item = self._process_text_reference(ref_num, data)
            page_num = data['texts'][ref_num]['prov'][0]['page_no']  # 从prov取页码

        elif ref_type == 'tables':                    # 处理表格引用
            content_item = {'type': 'table', 'table_id': ref_num}
            page_num = data['tables'][ref_num]['prov'][0]['page_no']

        elif ref_type == 'pictures':                  # 处理图片引用
            content_item = {'type': 'picture', 'picture_id': ref_num}
            page_num = data['pictures'][ref_num]['prov'][0]['page_no']

        # 将content_item归入对应页码
        if page_num not in pages:
            pages[page_num] = {'page': page_num, 'content': [], 'page_dimensions': bbox}
        pages[page_num]['content'].append(content_item)

    # 步骤3：按页码升序排列返回
    return [pages[pn] for pn in sorted(pages.keys())]
📖 核心概念详解
  • $ref 引用机制：Docling 用 "$ref": "#/texts/5" 引用内容而非直接嵌入。body记录"顺序"，texts数组记录"内容"，两者解耦
  • prov（provenance）溯源：每个text/table/picture都有 prov 字段，记录在PDF中的页码和坐标(bbox)。这是按页组织的唯一依据
  • 为什么用 dict 而不是 list 存 pages？因为页码可能不连续（如 {1:..., 3:..., 5:...}），用dict可以按页码直接索引，最后再sorted排序
📍 它在流程中的位置： 被 assemble_report() 调用。输出存入 JSON 的 content 字段。

⑱ assemble_tables() — 表格三格式输出 TABLES

🤔 这个函数做什么？（大白话）

遍历 Docling 识别出的每个表格，把每个表格输出为三种格式： ① 原始 JSON（完整数据）、② Markdown（给 LLM 用）、③ HTML（保留视觉结构）。 同时记录表格在 PDF 中的位置（页码、bbox坐标）和行列数。

🐍 关键代码 — 三格式输出 
for i, table in enumerate(tables):
    table_json_obj = table.model_dump()           # ① 原始JSON（完整数据）
    table_md = self._table_to_md(table_json_obj)  # ② Markdown（LLM友好）
    table_html = table.export_to_html()           # ③ HTML（渲染友好）

    table_obj = {
        'table_id': ref_num,      # 表格编号
        'page': table_page_num,   # 所在页码
        'bbox': [l, t, r, b],     # 边界框坐标
        '#-rows': nrows,          # 行数
        '#-cols': ncols,          # 列数
        'markdown': table_md,     # Markdown格式
        'html': table_html,       # HTML格式
        'json': table_json_obj    # 原始JSON
    }
💡 为什么输出三种格式？
- Markdown：直接塞进 LLM Prompt，token 效率最高
- HTML：保留合并单元格等视觉结构，适合前端渲染
- JSON：完整结构化数据，适合程序化数值提取
📍 它在流程中的位置： 被 assemble_report() 调用。输出存入 JSON 的 tables 字段。

⑲ _table_to_md() — Grid→Markdown转换 UTILITY

🤔 这个函数做什么？（大白话）

Docling 的表格数据存在 table['data']['grid'] 中——一个二维网格， 每个格子(cell)里有文字(text)。这个函数逐格提取文字，第一行当表头，其余当数据行， 用 tabulate 库渲染为 GitHub 风格的 Markdown 表格。

🐍 关键代码 — grid→text→Markdown 
def _table_to_md(self, table):
    table_data = []
    for row in table['data']['grid']:                    # 遍历每一行
        table_row = [cell['text'] for cell in row]       # 提取每格文字
        table_data.append(table_row)

    if len(table_data) > 1 and len(table_data[0]) > 0:   # 有表头的情况
        try:
            md_table = tabulate(table_data[1:],           # 数据行
                                headers=table_data[0],    # 表头
                                tablefmt="github")        # GitHub风格
        except ValueError:                                # 类型推断失败
            md_table = tabulate(table_data[1:], headers=table_data[0],
                                tablefmt="github", disable_numparse=True)  # 全当字符串
    else:
        md_table = tabulate(table_data, tablefmt="github")  # 无表头
    return md_table
📖 举例：输入→输出
  • 输入 grid：[[{text:"年份"}, {text:"营收"}], [{text:"2024"}, {text:"100亿"}]]
  • 输出 Markdown：
| 年份 | 营收  |
|------|-------|
| 2024 | 100亿 |
📍 它在流程中的位置： 被 assemble_tables() 调用，为每个表格生成 Markdown 格式。

⑳ assemble_pictures() — 图片信息组装 PICTURES

🤔 这个函数做什么？（大白话）

遍历 Docling 识别出的每张图片，提取图片位置（页码+bbox坐标）和关联文本（图注、说明文字）。 注意：图片的二进制数据不保存，只保存元数据和关联文本。

def assemble_pictures(self, data):

㉑ _process_picture_block() — 图片文本提取 UTILITY

🤔 这个函数做什么？（大白话）

Docling 把图片关联的文本（如图注 caption）放在 picture.children 中，通过 $ref 引用。 这个函数遍历这些引用，调用 _process_text_reference() 提取每个关联文本，复用了之前写好的文本处理逻辑。

🐍 关键代码
def _process_picture_block(self, picture, data):
    children_list = []
    for item in picture['children']:               # 遍历图片的子元素
        if isinstance(item, dict) and '$ref' in item:
            ref_type, ref_num = parse_ref(item['$ref'])
            if ref_type == 'texts':                 # 只处理文本引用
                content_item = self._process_text_reference(ref_num, data)
                children_list.append(content_item)
    return children_list
📍 它在流程中的位置： 被 assemble_pictures() 调用。内部复用了 _process_text_reference()。
