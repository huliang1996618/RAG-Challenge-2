① _process_chunk() — 模块级函数 PARALLEL HELPER

🤔 这个函数做什么？（大白话）

当启用并行模式时，主进程会把 PDF 文件分成好几堆（chunk）， 然后每堆交给一个子进程去处理。 这个函数就是子进程的"工作任务"——它在子进程里创建一个新的 PDFParser， 调用串行解析方法处理分配给它的那堆 PDF。

💡 为什么必须是模块级函数（不能写成类方法）？
  • Python 的 ProcessPoolExecutor 需要把任务函数 pickle（序列化） 后传给子进程
  • 类成员方法无法被 pickle，所以必须写成模块级的独立函数
  • 每个子进程有自己独立的 Python 解释器，所以可以各自创建自己的 PDFParser 实例
🐍 关键代码 
def _process_chunk(pdf_paths, pdf_backend, output_dir, num_threads, metadata_lookup, debug_data_path):
    """辅助函数：在独立的子进程中处理一批 PDF 文件"""
    parser = PDFParser(pdf_backend=pdf_backend, output_dir=output_dir,
                       num_threads=num_threads, csv_metadata_path=None)
    parser.metadata_lookup = metadata_lookup  # 从主进程"继承"元数据
    parser.debug_data_path = debug_data_path
    parser.parse_and_export(pdf_paths)        # 调用串行方法处理这批 PDF
    return f"Processed {len(pdf_paths)} PDFs."
📍 它在流程中的位置： 只在 parse_and_export_parallel() 中被调用 → 作为 executor.submit() 的任务函数

③ _parse_csv_metadata() — 静态方法 CSV

🤔 这个函数做什么？（大白话）

读一个 CSV 文件，里面有两列：sha1（PDF 的文件名哈希）和 company_name（公司名）。 把每一行转成 {sha1: {company_name: "某公司"}} 的字典， 这样后面处理 PDF 时就能根据文件名知道是哪个公司的年报。

🐍 关键代码 
@staticmethod
def _parse_csv_metadata(csv_path: Path) -> dict:
    import csv
    metadata_lookup = {}
    with open(csv_path, 'r', encoding='utf-8') as csvfile:
        reader = csv.DictReader(csvfile)  # 每行自动转为 {列名: 值}
        for row in reader:
            company_name = row.get('company_name', row.get('name', '')).strip('"')
            metadata_lookup[row['sha1']] = {'company_name': company_name}
    return metadata_lookup  # {"abc123...": {"company_name": "阿里巴巴"}}
📍 它在流程中的位置： 在 __init__ 中被调用（仅当传入了 CSV 路径时）。结果存入 self.metadata_lookup，后续 assemble_metainfo() 会用到。

④ _create_document_converter() — Docling 引擎配置 ENGINE

🤔 这个函数做什么？（大白话）

这是整个解析质量的"控制面板"。它创建一个 Docling 的 DocumentConverter 对象， 并配置好：用哪个 OCR、要不要识别表格、表格精度多高。 这些配置直接决定了解析出来的文字和表格准不准。

🐍 五个核心配置（逐一解释） 
pipeline_options = PdfPipelineOptions()
pipeline_options.do_ocr = True                           # ① 开启OCR，识别扫描件文字
ocr_options = EasyOcrOptions(lang=['en'], force_full_page_ocr=False)  # ② 仅英文，不全页OCR
pipeline_options.ocr_options = ocr_options
pipeline_options.do_table_structure = True               # ③ 检测PDF中的表格
pipeline_options.table_structure_options.do_cell_matching = True  # ④ 关联单元格和表头
pipeline_options.table_structure_options.mode = TableFormerMode.ACCURATE  # ⑤ 精度优先
📖 配置项详解（新手必读）
  • ① do_ocr=True：开启光学字符识别。如果PDF是扫描件（图片格式），没有这行就读不出文字
  • ② force_full_page_ocr=False：不全页OCR。Docling会先提取PDF中已嵌入的文字（如Word转的PDF），只有图片区域才用OCR。这样速度更快
  • ③ do_table_structure=True：检测PDF中哪里有表格，识别哪些单元格属于同一个表
  • ④ do_cell_matching=True：把表格单元格和它的表头对应起来（如知道"100亿"属于"2024年营收"列）
  • ⑤ ACCURATE模式：用TableFormer深度学习模型做表格识别，准确率高但速度较慢。年报财务报表精度比速度更重要
📍 它在流程中的位置：在 __init__ 中被调用，创建的 doc_converter 被 convert_documents() 使用。

⑤ convert_documents() — 批量转换 CONVERT

🤔 这个函数做什么？（大白话）

把一堆 PDF 文件路径扔给 Docling 引擎，引擎逐个解析，返回一个"结果流"。 注意它只是发起转换请求，真正的处理是惰性（lazy）的——当你开始遍历结果时才真正解析。

⑥ process_documents() — 处理转换结果 PROCESS

🤔 这个函数做什么？（大白话）

⑦ _normalize_page_sequence() — 页码规整 DEFENSE

🤔 这个函数做什么？（大白话）

Docling 有时会跳过完全空白的页面（比如年报中的空白分隔页），导致页码不连续。 这个函数检查"第1页到最后一页"之间是否缺页，缺了就补一个空页， 确保下游的 text_splitter 不会因为缺页编号而崩溃。

🐍 关键代码 
existing_pages = {page['page'] for page in data['content']}  # 收集所有存在的页码
max_page = max(existing_pages)                                # 最大页码
for page_num in range(1, max_page + 1):                       # 从1到最大页
    page_content = next(
        (page for page in data['content'] if page['page'] == page_num),  # 找这一页
        {"page": page_num, "content": [], "page_dimensions": {}}         # 找不到就补空页
    )
    new_content.append(page_content)
📍 它在流程中的位置： 在 process_documents() 中、export_to_dict() 之后立即调用。结果传给 assemble_report()。

⑧ parse_and_export() — 串行解析入口 ENTRY

🤔 这个函数做什么？（大白话）

这是串行模式的入口函数——把 PDF 文件一个接一个地解析。 它调用 convert_documents → process_documents 完成全流程。 如果任何一个 PDF 解析失败，抛异常停止（零容忍）。

🐍 关键代码 
def parse_and_export(self, input_doc_paths=None, doc_dir=None):
    start_time = time.time()                                # 开始计时
    if input_doc_paths is None and doc_dir is not None:
        input_doc_paths = list(doc_dir.glob("*.pdf"))       # 从目录收集PDF

    conv_results = self.convert_documents(input_doc_paths)  # 步骤1：Docling转换

    if failure_count > 0:
        raise RuntimeError(f"Failed converting {failure_count} documents.")
    _log.info(f"Completed in {elapsed_time:.2f} seconds.")

⑨ parse_and_export_parallel() — 并行解析入口 PARALLEL ENTRY

🤔 这个函数做什么？（大白话）

这是并行模式的入口——当 PDF 很多时（比如100份年报），串行太慢。 它把 PDF 列表切分成几个 chunk，用 ProcessPoolExecutor 启动多个独立子进程同时解析。每个子进程运行 _process_chunk()。

🐍 关键代码 — 三步并行策略 
# 步骤1：计算最优参数
cpu_count = multiprocessing.cpu_count()
optimal_workers = min(cpu_count, total_pdfs)          # 进程数 ≤ CPU核心数
chunk_size = max(1, total_pdfs // optimal_workers)     # 每个进程处理几个PDF

# 步骤2：切分PDF列表
chunks = [input_doc_paths[i:i+chunk_size] for i in range(0, total_pdfs, chunk_size)]

# 步骤3：提交到进程池并行执行
with ProcessPoolExecutor(max_workers=optimal_workers) as executor:
    futures = [executor.submit(_process_chunk, chunk, ...) for chunk in chunks]
    for future in as_completed(futures):
        result = future.result()  # 哪个chunk先完成就先处理哪个
📖 并行策略详解
  • optimal_workers = min(CPU核心数, PDF总数)：16核机器处理100个PDF → 16个进程，每个处理约6-7个
  • 为什么用多进程而不是多线程？PDF解析是CPU密集型，Python的GIL锁会让多线程等于单线程。多进程每个进程有独立Python解释器，真正并行
  • as_completed 实时追踪：不按提交顺序等待，哪个进程先跑完就先报告进度
📍 它在流程中的位置： 用户直接调用的并行入口。内部通过 ProcessPoolExecutor → _process_chunk → parse_and_export 完成解析。

⑪ assemble_report() — 报告组装总入口 ASSEMBLY

🤔 这个函数做什么？（大白话）

这是 JsonReportProcessor 的总调度函数——它不亲自干活， 而是调用5个子函数（metainfo、content、tables、pictures、debug）， 把各自的返回值拼成一个大字典。就像车间主任：分配任务给4个工人，最后汇总成果。

🐍 关键代码 — 五个组装步骤 
def assemble_report(self, conv_result, normalized_data=None):
    data = normalized_data if normalized_data is not None else conv_result.document.export_to_dict()
    assembled_report = {}
    assembled_report['metainfo']  = self.assemble_metainfo(data)   # ① 元数据
    assembled_report['content']   = self.assemble_content(data)    # ② 逐页内容
    assembled_report['tables']    = self.assemble_tables(          # ③ 表格
        conv_result.document.tables, data)
    assembled_report['pictures']  = self.assemble_pictures(data)   # ④ 图片
    self.debug_data(data)                                          # ⑤ 调试快照
    return assembled_report

⑫ assemble_metainfo() — 元数据提取 META

🤔 这个函数做什么？（大白话）

从 Docling 解析结果中数一数：有多少页、多少段文字、多少表格、多少图片。 然后从 CSV 元数据中查出公司名（通过 sha1 文件名匹配）。 这些信息存在 JSON 的 metainfo 字段中，供下游检索使用。

🐍 关键代码 
def assemble_metainfo(self, data):
    metainfo = {}
    sha1_name = data['origin']['filename'].rsplit('.', 1)[0]  # 去掉.pdf后缀
    metainfo['sha1_name'] = sha1_name
    metainfo['pages_amount'] = len(data.get('pages', []))        # 页数
    metainfo['text_blocks_amount'] = len(data.get('texts', []))   # 文本块数
    metainfo['tables_amount'] = len(data.get('tables', []))       # 表格数
    metainfo['pictures_amount'] = len(data.get('pictures', []))   # 图片数

    # 通过sha1文件名匹配CSV中的公司名
    if self.metadata_lookup and sha1_name in self.metadata_lookup:
        metainfo['company_name'] = self.metadata_lookup[sha1_name]['company_name']
    return metainfo
📍 它在流程中的位置： 被 assemble_report() 第一个调用。输出存入 JSON 的 metainfo 字段。

⑭ debug_data() — 调试输出 DEBUG

🤔 这个函数做什么？（大白话）

如果设置了 debug_data_path（调试路径），就把 Docling 的完整原始字典保存为 JSON 文件。 这是调试解析质量的"黄金数据源"——包含所有元数据、坐标、置信度等底层信息。 如果没设置调试路径，函数直接跳过，什么也不做。

🐍 关键代码 
def debug_data(self, data):
    if self.debug_data_path is None:              # 没配置调试路径 → 跳过
        return
    doc_name = data['name']
    path = self.debug_data_path / f"{doc_name}.json"
    path.parent.mkdir(parents=True, exist_ok=True)
    with path.open("w", encoding="utf-8") as f:
        json.dump(data, f, indent=2, ensure_ascii=False)  # 保存完整原始数据
📍 它在流程中的位置： 在 assemble_report() 最后调用，仅当用户配置了 debug_data_path 才生效。

⑮ expand_groups() — 展开分组结构 UTILITY

🤔 这个函数做什么？（大白话）

Docling 会把 PDF 中的列表项（如项目符号列表）放在 groups 数组中。 这个函数遍历 body.children，遇到 groups 类型的引用时， 把 group 里的每个子元素"拆出来"，"拍平"到顶层。 这样后续处理就不用管嵌套了，每个内容项都是平等的。

🐍 关键代码 
def expand_groups(self, body_children, groups):
    expanded_children = []
    for item in body_children:
        if isinstance(item, dict) and '$ref' in item:   # 是引用类型
            ref_type, ref_num = ref.split('/')[-2:]
            if ref_type == 'groups':                     # 是group引用
                group = groups[int(ref_num)]              # 取出group对象
                for child in group['children']:           # 遍历group的子元素
                    child_copy = child.copy()
                    child_copy['group_id'] = ref_num      # 标记来自哪个group
                    child_copy['group_name'] = group.get('name', '')
                    expanded_children.append(child_copy)   # "拍平"到顶层
            else:
                expanded_children.append(item)             # 非group引用保留
        else:
            expanded_children.append(item)                 # 非引用元素保留
    return expanded_children
📖 举例理解
  • 假设 body.children 中有 {$ref: "#/groups/0"}
  • groups[0] 包含3个子项：[$ref1, $ref2, $ref3]
  • expand_groups 把3个子项分别加入结果，并附带 group_id/group_name
  • 最终 expanded_children 中这3个子项和普通引用"平起平坐"
📍 它在流程中的位置： 被 assemble_content() 调用，作为其第一步。为后续的引用解析铺平道路。

⑯ _process_text_reference() — 文本引用处理 UTILITY

🤔 这个函数做什么？（大白话）

根据引用编号（如 #/texts/5 中的 5），从 data['texts'] 数组中取出第5个文本块， 然后构建一个标准化的内容字典：包含 text（文本内容）、type（是paragraph还是heading）、 text_id（编号，用于溯源）。如果 OCR 纠正了原文，还附带 orig 字段。

🐍 关键代码 
def _process_text_reference(self, ref_num, data):
    text_item = data['texts'][ref_num]               # 根据编号取文本块
    content_item = {
        'text': text_item.get('text', ''),            # 文本内容
        'type': text_item['label'],                    # 类型：paragraph/heading/footnote
        'text_id': ref_num                             # 编号（用于溯源）
    }
    # 如果OCR纠正了原文（orig≠text），附上原始内容
    if text_item.get('orig', '') != text_item.get('text', ''):
        content_item['orig'] = text_item['orig']
    # 附加上下文标记（列表编号、标记符号等）
    if 'enumerated' in text_item:
        content_item['enumerated'] = text_item['enumerated']   # 是否有序列表
    if 'marker' in text_item:
        content_item['marker'] = text_item['marker']           # 如 '•', '1.'
    return content_item
📖 返回示例
  • 普通段落：{text: "营收增长20%", type: "paragraph", text_id: 5}
  • OCR纠正后：{text: "Revenue", orig: "Revenuc", type: "paragraph", text_id: 12}
  • 列表项：{text: "第一步", type: "list_item", text_id: 8, marker: "1.", enumerated: True}
📍 它在流程中的位置： 被 assemble_content() 和 _process_picture_block() 调用。是文本标准化的核心函数。

⑰ assemble_content() — 按页组织内容 CORE

🤔 这个函数做什么？（大白话）

这是整个模块最复杂、最核心的函数。它把 Docling 分散在 body、texts、tables、pictures、groups 五个数组中的内容，按页码重新归类。最终输出 [{page:1, content:[...]}, {page:2, content:[...]}, ...]。

🐍 关键代码 — 三步走 
def assemble_content(self, data):
    pages = {}
    # 步骤1：展开groups（把嵌套列表"拍平"）
    expanded_body_children = self.expand_groups(
        data['body']['children'], data.get('groups', []))

    # 步骤2：遍历展开后的children，按$ref类型分发
    for item in expanded_body_children:
        ref_type, ref_num = parse_ref(item['$ref'])   # 解析 "#/texts/5" → ("texts", 5)

        if ref_type == 'texts':                       # 处理文本
            content_item = self._process_text_reference(ref_num, data)
            page_num = data['texts'][ref_num]['prov'][0]['page_no']  # 从prov取页码

        elif ref_type == 'tables':                    # 处理表格引用
            content_item = {'type': 'table', 'table_id': ref_num}
            page_num = data['tables'][ref_num]['prov'][0]['page_no']

        elif ref_type == 'pictures':                  # 处理图片引用
            content_item = {'type': 'picture', 'picture_id': ref_num}
            page_num = data['pictures'][ref_num]['prov'][0]['page_no']

        # 将content_item归入对应页码
        if page_num not in pages:
            pages[page_num] = {'page': page_num, 'content': [], 'page_dimensions': bbox}
        pages[page_num]['content'].append(content_item)

    # 步骤3：按页码升序排列返回
    return [pages[pn] for pn in sorted(pages.keys())]
📖 核心概念详解
  • $ref 引用机制：Docling 用 "$ref": "#/texts/5" 引用内容而非直接嵌入。body记录"顺序"，texts数组记录"内容"，两者解耦
  • prov（provenance）溯源：每个text/table/picture都有 prov 字段，记录在PDF中的页码和坐标(bbox)。这是按页组织的唯一依据
  • 为什么用 dict 而不是 list 存 pages？因为页码可能不连续（如 {1:..., 3:..., 5:...}），用dict可以按页码直接索引，最后再sorted排序
📍 它在流程中的位置： 被 assemble_report() 调用。输出存入 JSON 的 content 字段。

⑱ assemble_tables() — 表格三格式输出 TABLES

🤔 这个函数做什么？（大白话）

遍历 Docling 识别出的每个表格，把每个表格输出为三种格式： ① 原始 JSON（完整数据）、② Markdown（给 LLM 用）、③ HTML（保留视觉结构）。 同时记录表格在 PDF 中的位置（页码、bbox坐标）和行列数。

🐍 关键代码 — 三格式输出 
for i, table in enumerate(tables):
    table_json_obj = table.model_dump()           # ① 原始JSON（完整数据）
    table_md = self._table_to_md(table_json_obj)  # ② Markdown（LLM友好）
    table_html = table.export_to_html()           # ③ HTML（渲染友好）

    table_obj = {
        'table_id': ref_num,      # 表格编号
        'page': table_page_num,   # 所在页码
        'bbox': [l, t, r, b],     # 边界框坐标
        '#-rows': nrows,          # 行数
        '#-cols': ncols,          # 列数
        'markdown': table_md,     # Markdown格式
        'html': table_html,       # HTML格式
        'json': table_json_obj    # 原始JSON
    }
💡 为什么输出三种格式？
- Markdown：直接塞进 LLM Prompt，token 效率最高
- HTML：保留合并单元格等视觉结构，适合前端渲染
- JSON：完整结构化数据，适合程序化数值提取
📍 它在流程中的位置： 被 assemble_report() 调用。输出存入 JSON 的 tables 字段。

⑲ _table_to_md() — Grid→Markdown转换 UTILITY

🤔 这个函数做什么？（大白话）

Docling 的表格数据存在 table['data']['grid'] 中——一个二维网格， 每个格子(cell)里有文字(text)。这个函数逐格提取文字，第一行当表头，其余当数据行， 用 tabulate 库渲染为 GitHub 风格的 Markdown 表格。

🐍 关键代码 — grid→text→Markdown 
def _table_to_md(self, table):
    table_data = []
    for row in table['data']['grid']:                    # 遍历每一行
        table_row = [cell['text'] for cell in row]       # 提取每格文字
        table_data.append(table_row)

    if len(table_data) > 1 and len(table_data[0]) > 0:   # 有表头的情况
        try:
            md_table = tabulate(table_data[1:],           # 数据行
                                headers=table_data[0],    # 表头
                                tablefmt="github")        # GitHub风格
        except ValueError:                                # 类型推断失败
            md_table = tabulate(table_data[1:], headers=table_data[0],
                                tablefmt="github", disable_numparse=True)  # 全当字符串
    else:
        md_table = tabulate(table_data, tablefmt="github")  # 无表头
    return md_table
📖 举例：输入→输出
  • 输入 grid：[[{text:"年份"}, {text:"营收"}], [{text:"2024"}, {text:"100亿"}]]
  • 输出 Markdown：
| 年份 | 营收  |
|------|-------|
| 2024 | 100亿 |
📍 它在流程中的位置： 被 assemble_tables() 调用，为每个表格生成 Markdown 格式。

⑳ assemble_pictures() — 图片信息组装 PICTURES

🤔 这个函数做什么？（大白话）

遍历 Docling 识别出的每张图片，提取图片位置（页码+bbox坐标）和关联文本（图注、说明文字）。 注意：图片的二进制数据不保存，只保存元数据和关联文本。

🐍 关键代码
def assemble_pictures(self, data):
    assembled_pictures = []
    for picture in data['pictures']:
        children_list = self._process_picture_block(picture, data)  # 提取图注文本

        picture_obj = {
            'picture_id': ref_num,          # 图片编号
            'page': picture_page_num,       # 所在页码
            'bbox': [l, t, r, b],           # 边界框坐标
            'children': children_list,      # 关联文本（如图注）
        }
        assembled_pictures.append(picture_obj)
    return assembled_pictures
⚠️ 注意： 图片二进制数据不保存在JSON中。仅保存位置信息和关联文本。实际图像需另外提取。
📍 它在流程中的位置： 被 assemble_report() 调用。输出存入 JSON 的 pictures 字段。

㉑ _process_picture_block() — 图片文本提取 UTILITY

🤔 这个函数做什么？（大白话）

Docling 把图片关联的文本（如图注 caption）放在 picture.children 中，通过 $ref 引用。 这个函数遍历这些引用，调用 _process_text_reference() 提取每个关联文本，复用了之前写好的文本处理逻辑。

🐍 关键代码
def _process_picture_block(self, picture, data):
    children_list = []
    for item in picture['children']:               # 遍历图片的子元素
        if isinstance(item, dict) and '$ref' in item:
            ref_type, ref_num = parse_ref(item['$ref'])
            if ref_type == 'texts':                 # 只处理文本引用
                content_item = self._process_text_reference(ref_num, data)
                children_list.append(content_item)
    return children_list
📍 它在流程中的位置： 被 assemble_pictures() 调用。内部复用了 _process_text_reference()。

🛠️ 九、辅助机制 — 页码规整 & 调试快照

📏 _normalize_page_sequence

遍历现有页码，找出最大页码。对于 1 到 max_page 之间缺失的页码，插入空页面模板（content: [], page_dimensions: {}）。确保下游 text_splitter 不会因页码不连续而出错。

🐛 debug_data

当 debug_data_path 被设置时，将 Docling 的完整原始字典保存为 JSON 文件。这是调试解析质量的"黄金数据源"——包含所有元数据、坐标、置信度等信息。

new_content = []
