🔥 Q1：为什么有三个 Retriever 类，而不是一个？

BM25Retriever、VectorRetriever、HybridRetriever 各自独立实现了 retrieve_by_company_name()。它们分别解决什么问题？为什么不用一个统一的 Retriever 基类 + 策略模式？

💡 提示：看看每个类的 __init__ 需要什么资源（BM25 索引、FAISS 索引、LLMReranker）。它们的"重量"相同吗？

📝 点击查看答案
三个 Retriever 的"重量级"差异巨大——统一基类反而增加耦合。看各自的构造器：

BM25Retriever — 最轻量，纯文件IO 
class BM25Retriever:
    def __init__(self, bm25_db_dir: Path, documents_dir: Path):
        self.bm25_db_dir = bm25_db_dir      # ← *.pkl 文件目录
        self.documents_dir = documents_dir   # ← *.json 文档目录
        # 无 API Key、无模型加载、无内存预热
        # 检索时才 load pickle 文件
BM25 索引是纯权重矩阵（几十MB），不需要网络，不需要 GPU。每次查询 retrieve_by_company_name() 内部才 pickle.load() 加载对应公司的索引。

VectorRetriever — 中等重量，需要 API + 内存预加载 
class VectorRetriever:
    def __init__(self, vector_db_dir, documents_dir):
        self.vector_db_dir = vector_db_dir
        self.documents_dir = documents_dir
        self.all_dbs = self._load_dbs()   # ← 启动时加载所有 FAISS 索引到内存
        self.llm = self._set_up_llm()     # ← 创建 OpenAI 客户端（API Key）
_load_dbs() 在 构造时 就扫描两个目录，按文件名匹配 报告A.json ↔ 报告A.faiss，全部读入内存。如果有 50 家公司报告，构造时就把 50 个 FAISS 索引全部加载完——启动慢但检索快。

HybridRetriever — 最重量级，组合向量检索 + LLM 精排 
class HybridRetriever:
    def __init__(self, vector_db_dir, documents_dir):
        self.vector_retriever = VectorRetriever(vector_db_dir, documents_dir)
        self.reranker = LLMReranker()  # ← 又一个 OpenAI 客户端 + prompt 模板
LLMReranker 内部也创建了 OpenAI()，所以 HybridRetriever 实际持有 两个 OpenAI 客户端（VectorRetriever 的 + Reranker 的）。这还不是最重的——每次检索会额外调用 14~28 次 GPT-4o-mini API（见 Q3）。

为什么不用基类 + 策略模式？
如果写一个 class Retriever(ABC)，强制要求 retrieve_by_company_name() 签名一致，会导致：
① BM25Retriever 被迫持有 llm 属性（用不到）→ 空依赖污染
② HybridRetriever 被迫接受 bm25_db_dir 参数（用不到）→ 参数冗余
③ 三者的 distance 字段含义完全不同：BM25=词频分、向量=余弦相似度、Hybrid=LLM+向量融合分 → 统一接口反而误导下游消费者

实际做法是"组合 + 独立接口"：VectorRetriever 被 HybridRetriever 组合使用，BM25Retriever 独立存在。这不是设计模式的失败，而是对"不必要的抽象"的克制。

🔥 Q2：HybridRetriever 为什么不用 BM25？

HybridRetriever 的第一阶段只调用了 VectorRetriever，完全没有用到 BM25Retriever。BM25Retriever 和 BM25Ingestor（在 ingestion.py 中）都已经完整实现了——那它存在的意义是什么？一个真正的 "Hybrid"（混合检索）应该是 BM25 + 向量双路召回再融合，但这里却绕过了 BM25。这是设计疏忽还是有意为之？

💡 提示：看看 reranker.rerank_documents() 的输入输出格式和 BM25Retriever 的返回格式，两者能直接对接吗？

📝 点击查看答案
HybridRetriever 只用了 Vector 作为第一阶段召回，BM25 被搁置。这是三个原因叠加的结果：

看 HybridRetriever.retrieve_by_company_name() 的实际代码： 
def retrieve_by_company_name(self, company_name, query,
    llm_reranking_sample_size=28, documents_batch_size=2, top_n=6,
    llm_weight=0.7, return_parent_pages=False):

    # 第一阶段：只用向量检索（完全不碰 BM25）
    vector_results = self.vector_retriever.retrieve_by_company_name(
        company_name=company_name, query=query,
        top_n=llm_reranking_sample_size,  # 28 个候选
        return_parent_pages=return_parent_pages
    )

    # 第二阶段：LLM 精排
    reranked_results = self.reranker.rerank_documents(
        query=query, documents=vector_results,
        documents_batch_size=documents_batch_size, llm_weight=llm_weight
    )
    return reranked_results[:top_n]  # 取前 6
原因 1 — BM25 返回格式不兼容：
BM25Retriever 返回的是裸 dict： 
[{"distance": 7.23, "page": 42, "text": "公司营收..."}]
而 reranker.rerank_documents() 需要从 dict 中取 doc['text'] 作为 LLM 评分输入、用 doc['distance'] 做融合。BM25 的 distance（词频分，范围 0~N）和向量的 distance（余弦相似度，范围 0~1）量纲完全不同——直接融合会炸掉排序。

原因 2 — 中文分词完全失效（详见 Q8）：
BM25 两端都用 query.split() / chunk.split() 按空格分词。中文没有空格： 
"比亚迪2024年营收增长率".split() → ["比亚迪2024年营收增长率"]  # 一个"词"
# BM25 索引里所有文档也都是 split() 切出来的，根本匹配不上
实际召回率为零。需要接 jieba 分词才能工作，但当前代码未实现。

原因 3 — 两阶段策略已经足够：
向量召回 28 个 → LLM 精排 6 个 的流水线在实践中已经产生高质量结果。BM25 在此架构中更多是独立对比基线和未来扩展点——如果需要真正的 Hybrid（双路召回+LLM精排），只需在 BM25 加 jieba 分词后，将两路结果合并再送入 reranker。

⚡ Q3：documents_batch_size=1 vs =2，发生了什么不同的事？

当 batch_size=1 时，每个文档单独调用一次 LLM；当 batch_size=2 时，两个文档打包进一次 LLM 调用。前者用 ThreadPoolExecutor 并行发多个请求，后者也并行但每个请求里评两个文档。哪种更快？哪种更准？为什么默认值是 2 而不是 4 或 8？

💡 提示：想想 LLM 的 context window 和 attention 机制——同时看两个文档和同时看八个文档，LLM 的"比较能力"有区别吗？

📝 点击查看答案
batch_size 控制 LLM 在一次 API 调用中评估几个文档。看 rerank_documents() 的分支逻辑：

分支 1 — batch_size=1：独立评估，ThreadPoolExecutor 并行 
if documents_batch_size == 1:
    def process_single_doc(doc):
        ranking = self.get_rank_for_single_block(query, doc['text'])
        # get_rank_for_single_block 内部：
        #   user_prompt = f'Here is the query:\n"{query}"\n\n
        #                   Here is the retrieved text block:\n"""{retrieved_document}"""\n'
        #   → LLM 只看这一个文档，回答 relevance_score + reasoning
        doc_with_score = doc.copy()
        doc_with_score["relevance_score"] = ranking["relevance_score"]
        doc_with_score["combined_score"] = round(
            llm_weight * ranking["relevance_score"] + vector_weight * doc['distance'], 4)
        return doc_with_score

    # 28 个文档 → 28 次 API 调用，全部并行发出
    with ThreadPoolExecutor() as executor:
        all_results = list(executor.map(process_single_doc, documents))
分支 2 — batch_size=2（默认）：配对评估，批次内并行 
else:  # batch_size > 1
    # 28 个文档 → ceil(28/2) = 14 个 batch
    doc_batches = [documents[i:i+2] for i in range(0, len(documents), 2)]

    def process_batch(batch):
        texts = [doc['text'] for doc in batch]
        rankings = self.get_rank_for_multiple_blocks(query, texts)
        # get_rank_for_multiple_blocks 内部：
        #   formatted = "Block 1:\n\n"""\n{doc1}\n"""\n\n---\n\nBlock 2:\n\n"""\n{doc2}\n""""
        #   user_prompt = f'Here is the query: "{query}"\n\n{formatted}\n\n
        #                   You should provide exactly 2 rankings, in order.'
        #   → LLM 同时看 Block 1 和 Block 2，分别打分
        for doc, rank in zip(batch, block_rankings):
            doc["combined_score"] = round(
                llm_weight * rank["relevance_score"] + vector_weight * doc['distance'], 4)

    with ThreadPoolExecutor() as executor:
        batch_results = list(executor.map(process_batch, doc_batches))
batch_size=2 为什么是最优默认值？

① 减少网络往返：28 个候选 → 14 次 API 请求（vs batch_size=1 的28次），RTT 减半
② LLM 能配对比较：当 LLM 同时看到 Block A 和 Block B，它能说"A 比 B 更相关"，而不仅仅各自打分。这是 LLM 擅长的任务
③ 避免注意力稀释：如果 batch_size=8，LLM 一次看8个文档，中间位置的文档容易被忽略（Lost-in-the-Middle 效应）。2 是最小比较单元，注意力最集中
④ 降级容错：代码有防御： 
if len(block_rankings) < len(batch):
    # LLM 少返回了排名 → 用 0.0 补齐
    for _ in range(len(batch) - len(block_rankings)):
        block_rankings.append({"relevance_score": 0.0, ...})
不会因为 LLM 少输出一个排名而崩溃。

📊 Q4：llm_weight=0.7 意味着什么？

最终得分 = 0.7 × LLM相关性分 + 0.3 × 向量相似度。为什么要给 LLM 70% 的权重？如果改成 0.0（纯向量）或 1.0（纯 LLM），检索结果会怎样变化？0.7 这个值是从哪里来的——是调参调出来的，还是有理论依据？

💡 提示：回顾 reranking.py 中 LLM 用的是 gpt-4o-mini，向量用的是 text-embedding-3-large。两个模型的"判断力"真的差 7:3 吗？

📝 点击查看答案
llm_weight 控制 LLM 精排话语权 vs 向量基础信号的权重分配。核心公式：

vector_weight = 1 - llm_weight  # 0.3
combined_score = round(
    llm_weight * relevance_score +      # 0.7 × GPT-4o-mini 判断
    vector_weight * distance,            # 0.3 × FAISS 内积相似度
    4
)
三个极端值的效果分析：

llm_weight=0.0（纯向量排序）： 
combined_score = 0.0 × relevance_score + 1.0 × distance  # 完全是向量距离
# 查询"原油进口量" → 向量空间中 "原油"≈"石油"≈"天然气"≈"成品油"
# embedding 对这四个词给出几乎相同的 cosine similarity
# 结果：排序混乱，真正包含"原油进口量"数据的 chunk 可能排在第 5 位
向量擅长"语义近似"但不擅长"精确区分"——它无法判断"相似"是"信息相关"还是"语义重叠但无关"。

llm_weight=1.0（纯 LLM 排序）： 
combined_score = 1.0 × relevance_score + 0.0 × distance
# 每个 query 需要 28 次 LLM 调用（未预筛选）
# 成本：28 × GPT-4o-mini API × 每次 ~200 token output ≈ 5600 token
# 延迟：虽然有 ThreadPoolExecutor 并行，但 28 个请求仍然比向量检索慢 100×
更重要的是：完全不用向量预筛选，LLM 需要评估海量不相关文档。向量先做粗筛去掉 99% 的不相关内容，LLM 只在 28 个候选上精排——分工明确。

llm_weight=0.7（默认）：
0.7 不是理论推导值，而是实践中的经验选择。逻辑是：
① GPT-4o-mini 阅读文本的能力明显优于 text-embedding-3-large 的向量空间近似 → 给 LLM 更大话语权
② 保留 0.3 向量分量作为"锚点"——防止 LLM 对某文档误判为极度相关（比如被一段写得特别好的文字"忽悠"了），向量分量提供了基础信号约束
③ 单次 LLM 评估可能有随机性（即使 temperature=0），0.3 向量分提供了一定的平滑

📊 Q5：为什么要召回 28 个再筛选到 6 个？

llm_reranking_sample_size=28，top_n=6。也就是先从向量库召回 28 个候选，用 LLM 重排序后再取前 6 个。为什么不在向量检索时直接 top_n=6？省掉的那 22 个候选文档会造成什么损失？

💡 提示：想想向量检索的"语义相近但无关"问题——一个 query 关于"营收增长"，向量库可能把"净利润增长"的 chunk 也排进前 6，因为它们在 embedding 空间里确实很近。LLM 能纠正这个错误吗？

📝 点击查看答案
28→6 是经典的两阶段"召回-重排"（Recall-Rerank）策略。看 HybridRetriever 的完整流程：

# 阶段 1：向量粗筛（高召回，容忍噪声）
vector_results = self.vector_retriever.retrieve_by_company_name(
    company_name=company_name, query=query,
    top_n=28,  # ← 28 不是随意选的，是召回率与成本之间的权衡
    return_parent_pages=False
)
# FAISS 返回 28 个结果，每个都带有 cosine similarity（0~1）

# 阶段 2：LLM 精排（高精度，筛选无关）
reranked_results = self.reranker.rerank_documents(
    query=query, documents=vector_results,  # ← 只看这28个
    documents_batch_size=2, llm_weight=0.7
)
return reranked_results[:6]  # ← 最终只取前6
为什么不能直接 top_n=6（省掉 22 个候选）？——具体例子说明：

查询："比亚迪 2024 年营收增长率是多少？"

向量检索 top-6 的实际结果可能是：
排名chunk 内容余弦相似度实际相关？
1"2024年营收达到6000亿元，同比增长12%"0.91✅
2"净利润同比增长15%至300亿元"0.88❌ 问的是营收不是利润
3"2024年营收目标为6500亿元"0.87❌ 目标是目标，不是实际增长率
4"2024年营收构成：汽车业务占80%"0.86❌ 说的是构成不是增长率
5"2023年营收5000亿元"0.85❌ 2023年数据，不能直接算增长率
6"主要客户营收贡献情况"0.84❌ 客户维度，不是总营收
只有第1个相关！如果直接 top_n=6，LLM 只有一个正确答案可用。但第 9 位可能是： 
排名9: "营收从2023年的5000亿增长到2024年的6000亿，增长率达12%"
余弦相似度: 0.81  → 因为用了"增长到"而非"增长率"，embedding 给低了分
这个 chunk 比排名2-6都更相关，但它排在第9位。top_n=28 确保了这些"语义不够近但信息高度相关"的文档能被 LLM 发现。 28 这个数字怎么来的？实践中的经验值：太小（如 10）→ 可能漏掉正确答案；太大（如 50）→ LLM 成本线性增长。28 ≈ 向量检索中 90%+ 召回率覆盖的最小 k 值。

📊 Q6：return_parent_pages — chunk 和整页的区别是什么？

return_parent_pages=False 返回的是切分后的 300-token chunk；=True 返回的是 chunk 所属的完整页面（可能几千 tokens）。下游什么时候需要 chunk，什么时候需要整页？如果用户问"比亚迪 2024 年利润表"，返回 chunk 够用吗？返回整页会不会给 LLM 太多噪音？

💡 提示：看看 HybridRetriever 的默认值是 return_parent_pages=False——说明这个 pipeline 的"主流用法"是返回 chunk。谁在消费 chunk？谁需要整页？

📝 点击查看答案
return_parent_pages 控制返回粒度：chunk（300 token 片段）vs 完整页面（几千 token）。看检索代码中的分支：

for distance, index in zip(distances[0], indices[0]):  # FAISS 返回的 top-N
    distance = round(float(distance), 4)
    chunk = chunks[index]                             # ← 切分后的小块
    parent_page = next(                               # ← chunk 所属的完整页面
        page for page in pages if page["page"] == chunk["page"]
    )

    if return_parent_pages:
        if parent_page["page"] not in seen_pages:     # ← 去重！
            seen_pages.add(parent_page["page"])
            result = {
                "distance": distance,                  # 保留了 chunk 的相似度
                "page": parent_page["page"],           # 但返回整页的内容
                "text": parent_page["text"]             # ← 整页文本（可能数千字符）
            }
    else:
        result = {                                     # 默认：返回 chunk 自身
            "distance": distance,
            "page": chunk["page"],
            "text": chunk["text"]                      # ← 仅 300 token
        }
关键细节——去重（seen_pages）：
一个页面可能被切成 5~10 个 chunk。如果其中 3 个 chunk 都匹配了 query，不拿去重就会返回同一页 3 次。去重确保每页最多返回一次。

两种场景对应两种需求：

Chunk 模式（False）— 精确回答
Query："比亚迪 2024 年研发投入多少钱？" → 答案就在一两句话里 → chunk 精准命中： 
返回: {"text": "2024年度研发投入为150亿元，同比增长20%", "page": 42, "distance": 0.93}
# 150 个 token，LLM 一眼看到答案
整页模式（True）— 完整理解
Query："分析比亚迪 2024 年利润表的整体状况" → 需要利润表全部行 + 上下文 → 整页返回： 
返回: {"text": "第42页完整文本（含利润表HTML + 3段文字说明）共2800字符", "page": 42}
# 虽然 90% 可能与问题无关，但 LLM 需要完整利润表才能正确分析
噪音 vs 完整性 的权衡：
整页模式可能返回 2000+ token，其中大量内容不相关。这违背了 RAG "只给 LLM 最相关信息"的核心原则。默认值 False 说明 pipeline 的主流用法是 chunk 检索——精确高效。整页模式是为需要完整上下文的分析型 query 准备的例外开关。

🔥 Q7：retrieve_all() 为什么给所有页面都标 distance=0.5？

这个方法返回公司的所有页面，每页的 distance 硬编码为 0.5。它的使用场景是什么？如果用户想"看完比亚迪的整份年报"，为什么不直接用 retrieve_all() 而要绕一圈？如果把它返回的结果直接喂给 LLMReranker.rerank_documents()，会发生什么？

📝 点击查看答案
retrieve_all() 是一个"放弃检索，全量返回"的兜底策略。看代码：

def retrieve_all(self, company_name: str) -> List[Dict]:
    # 第一步：找到目标公司（和正常检索一样）
    for report in self.all_dbs:
        if report["document"]["metainfo"]["company_name"] == company_name:
            target_report = report
            break

    pages = target_report["document"]["content"]["pages"]

    # 第二步：不做任何检索，返回所有页面
    all_pages = []
    for page in sorted(pages, key=lambda p: p["page"]):  # 按页码排序
        result = {
            "distance": 0.5,       # ← 硬编码！所有页面同权重
            "page": page["page"],
            "text": page["text"]
        }
        all_pages.append(result)
    return all_pages
distance=0.5 的三个设计意图：

① 不是 0：如果 distance=0，下游可能触发距离阈值过滤（距离太小→不相关→丢弃）。0.5 保证不会被过滤
② 不是 1：如果 distance=1，下游可能误以为这是"完美语义匹配"，做出错误推断。0.5 明确表达"没有检索信号"
③ 中性的 0.5：在排序中不会"挤掉"真正检索到的结果（检索结果一般在 0.7~0.95），也不会被它们完全淹没

如果喂给 LLMReranker——灾难性的连锁反应： 
# 假设年报有 200 页，全量给 reranker
vector_results = retrieve_all("比亚迪")  # 返回 200 个 page
reranked_results = reranker.rerank_documents(
    query="研发投入多少？", documents=vector_results,  # 200 个文档！
    documents_batch_size=2  # ceil(200/2) = 100 次 LLM 调用！
)
# 成本：100 × GPT-4o-mini ≈ 5000 token 输出 + 200页×平均500 token 输入 = 10万 token
# 延迟：100 次 API 调用即使全并行也需 10-30 秒
# 准确度：LLM 在 200 个候选间排序 → Lost-in-the-Middle 效应严重
正确的全量分析方式：先用 retrieve_all() 获取所有页面 → 按页号天然顺序排列 → 分批（每 10 页）输入 LLM 做摘要 → 汇总答案。这样 LLM 不需要在 200 个候选间竞争排序，只需要按序理解内容。

⚡ Q8：BM25 的 query.split() 对中文管用吗？

Ingestion 和 Retrieval 两端都是简单的 .split()（按空格切分）。中文没有空格分隔——查询"比亚迪2024年营业收入"会被切成什么？BM25 索引到底能不能工作？如果不能，应该怎么改？

💡 提示：这和 Q2 有关——也许 BM25 之所以没被 HybridRetriever 使用，就是因为中文分词没做好？

📝 点击查看答案
BM25 的中文分词完全失效——这是它没有被集成到 HybridRetriever 的根本原因。看两端的代码：

Ingestion 端（建索引时）： 
# ingestion.py — BM25Ingestor.create_bm25_index()
def create_bm25_index(self, chunks: List[str]) -> BM25Okapi:
    tokenized_chunks = [chunk.split() for chunk in chunks]  # ← .split()!
    return BM25Okapi(tokenized_chunks)
# 中文 chunk "比亚迪2024年实现营业收入6000亿元".split()
# → ["比亚迪2024年实现营业收入6000亿元"]   ← 整个是一个"token"！
Retrieval 端（查询时）： 
# retrieval.py — BM25Retriever.retrieve_by_company_name()
tokenized_query = query.split()   # ← 又 .split()！
scores = bm25_index.get_scores(tokenized_query)
# "比亚迪2024年营收".split()
# → ["比亚迪2024年营收"]   ← 又是一个"token"
为什么永远匹配不上？BM25Okapi 的 get_scores() 做的是精确 token 匹配：query 的 token 必须出现在文档的 token 列表中才能得分。两端都是 .split()，中文被切成单个大字符串，索引中的 "比亚迪2024年实现营业收入6000亿元" 和 query 的 "比亚迪2024年营收" 是两个不同的字符串——零匹配，零分。

正确做法——接入 jieba 分词： 
import jieba

# Ingestion 端
tokenized_chunks = [list(jieba.cut(chunk)) for chunk in chunks]
# "比亚迪2024年实现营业收入6000亿元"
# → ["比亚迪", "2024年", "实现", "营业", "收入", "6000", "亿", "元"]

# Retrieval 端
tokenized_query = list(jieba.cut(query))
# "比亚迪2024年营收"
# → ["比亚迪", "2024年", "营收"]
# "营收" 能匹配索引中的 "收入"？不，jieba 不会同义词扩展...
# 需要额外加同义词词典：{"营收": ["收入", "营业收入"]}
即使加上 jieba，BM25 对于中文还有一个挑战：同义词。"营收" 和 "营业收入" 在 BM25 看来是完全不同的词（需要同义词扩展），而向量 embedding 天然能处理同义词。这也是向量检索在这个项目中成为主流方案的原因之一。

这也回答了 Q2：BM25 没进 HybridRetriever 不全是设计问题，更是工程现实——中文 BM25 需要 jieba + 同义词词典才能生产可用，当前代码还没做到这一步。

🔥 Q10：FAISS 用的是内积（Inner Product）还是余弦距离？

ingestion.py 第 85 行：index = faiss.IndexFlatIP(dimension) # Cosine distance。注释写的是"Cosine distance"，但实际用的是 IndexFlatIP（Inner Product，内积）。内积和余弦距离是两个不同的度量——这个注释是对的还是错的？对于 OpenAI text-embedding-3-large 输出的 embedding（是归一化的吗？），内积和余弦相似度等价吗？更重要的是——combined_score = 0.7×LLM + 0.3×distance 中，直接把 inner product 分数（越大越好）和 relevance_score（越大越好）相加——这个融合在数学上合理吗？如果 FAISS 用的是 L2 距离（越小越好），这行代码会不会把排序完全搞反？

💡 提示：faiss.IndexFlatIP 返回的 distances 是内积值（范围 [-∞, +∞]），而 LLMReranker 返回的 relevance_score 可能范围不同（看看 prompts.py）。两个量纲不同的分数直接相加，是否应该先归一化？

📝 点击查看答案
这是整个项目中最微妙的数学细节。先从 ingestion.py 的代码看起：

# ingestion.py — VectorDBIngestor._create_vector_db()
def _create_vector_db(self, embeddings: List[float]):
    embeddings_array = np.array(embeddings, dtype=np.float32)
    dimension = len(embeddings[0])          # text-embedding-3-large → 3072
    index = faiss.IndexFlatIP(dimension)     # Inner Product（内积），不是 L2！
    #        ↑↑↑↑↑↑↑↑↑↑↑↑
    # 注释写的是 "# Cosine distance"，但实际用的是 IndexFlatIP
    index.add(embeddings_array)
    return index
注释"Cosine distance"是对的还是错的？——数学上是正确的！

关键：OpenAI text-embedding-3-large 输出的 embedding 是L2 归一化的（已验证，||v||₂ ≈ 1.0）。

归一化向量的数学恒等式： 
余弦相似度 cos(θ) = (v · q) / (||v|| · ||q||)
当 ||v|| = ||q|| = 1（归一化）:
    cos(θ) = v · q   ← 内积 = 余弦相似度！

IndexFlatIP 计算的就是 v · q（内积）
∴ 对归一化向量，IndexFlatIP 返回的值 = 余弦相似度 ∈ [-1, 1]
实际中，text-embedding-3-large 的向量集中在正象限，余弦相似度通常在 [0, 1]，所以注释 "Cosine distance" 在效果上完全等价。严格来说应该写 "Cosine similarity via Inner Product for normalized vectors"，但注释的本意是正确的。

分数融合的合理性分析： 
combined_score = 0.7 * relevance_score + 0.3 * distance
# relevance_score: GPT-4o-mini 返回，通过 response_format 约束，范围 [0, 1]
# distance: FAISS 内积值，归一化后范围 [0, 1]（实际通常 0.7~0.95）
# → 两个量纲都在 [0, 1]，可以直接相加！
潜在风险——LLM relevance_score 的量纲不确定： 
# prompts.py 中 RerankingPrompt 对 relevance_score 的描述：
# "介于 0（完全不相关）到 1（高度相关）之间的浮点数"
# 但如果 LLM 实际返回的值集中在 0.3~0.8（而非均匀分布在 0~1），
# 那么 0.7×LLM 分 + 0.3×FAISS 分的融合就会偏斜
如果用了 IndexFlatL2 会怎样？——排序完全反转！ 
# L2 距离：越小越好（0=完全相同，∞=完全不同）
# 假设检索返回：
chunk_A: L2=0.15 (很相似), relevance=0.9 → combined = 0.7×0.9 + 0.3×0.15 = 0.675
chunk_B: L2=0.85 (很不像), relevance=0.1 → combined = 0.7×0.1 + 0.3×0.85 = 0.325
# 结果还是 A > B？看似正确...

# 但在边缘情况下：
chunk_C: L2=0.10 (最相似！), relevance=0.2 → combined = 0.7×0.2 + 0.3×0.10 = 0.170
chunk_D: L2=0.90 (很不像),   relevance=0.9 → combined = 0.7×0.9 + 0.3×0.90 = 0.900
# C 的 L2 距离最小（最相似）但总分 0.17，D 的 L2 最大（最不相似）但总分 0.90！
# 排序完全反了！需要用 1/(1+L2) 或 exp(-L2) 做方向转换。
这就是为什么 IndexFlatIP（内积，越大越好）是正确选择——它和 relevance_score 的方向一致（都是越大越好），可以直接相加。如果用 L2，需要方向转换逻辑，当前代码未实现这个转换。

① BM25Retriever — 关键词检索 SPARSE

🤔 这个类是做什么的？（大白话）

"找包含这些词的文档"——BM25 是最经典的关键词检索算法。它统计 query 中的词在每篇文档中出现了多少次， 给一个分数。词出现越多、越稀有，分数越高。简单粗暴，但无法理解"营收"和"收入"是近义词。

📋 唯一方法：retrieve_by_company_name()

参数：company_name（公司名）、query（问题）、top_n（返回条数）、return_parent_pages（返回整页而非 chunk）

1
遍历 documents_dir 下所有 JSON，找到 company_name 匹配的报告
2
从 bm25_db_dir 加载对应的 .pkl BM25 索引文件
3
对 query 分词（query.split()），调用 bm25_index.get_scores() 获取每个 chunk 的 BM25 分数
4
按分数降序排列，取前 top_n 个，返回 {distance, page, text}
🐍 BM25 检索核心代码 
scores = bm25_index.get_scores(tokenized_query)       # 每个chunk的BM25分数
top_indices = sorted(range(len(scores)),
    key=lambda i: scores[i], reverse=True)[:top_n]    # 取Top-N

for index in top_indices:
    chunk = chunks[index]
    result = {"distance": score, "page": chunk["page"], "text": chunk["text"]}
⚠️ 当前代码中 BM25Retriever 未被 HybridRetriever 使用。 从代码看，HybridRetriever 只组合了 VectorRetriever + LLMReranker，BM25 作为一个独立备选方案存在。

② VectorRetriever — 向量语义检索 CORE

🤔 这个类是做什么的？（大白话）

"找语义最相似的文档"——它把 query 变成一个向量（用 OpenAI embedding），然后在 FAISS 索引里找余弦距离最近的几个 chunk。 它能理解"营收"≈"收入"≈"营业收入"，因为它们的向量在空间中很接近。

📐 构造函数做了什么？

初始化三步走
  1. _load_dbs()：批量加载所有 FAISS 索引文件 + 对应文档 JSON
  2. _set_up_llm()：创建 OpenAI 客户端，用于后续生成 embedding
  3. 把每个报告的 {name, vector_db, document} 存入 self.all_dbs

📦 _load_dbs() — 加载所有向量索引

扫描 documents_dir 和 vector_db_dir 两个目录， 按文件名匹配：报告A.json ↔ 报告A.faiss。加载成功的数据结构：

📊 self.all_dbs 数据结构 
[
  {
    "name": "abc123...sha1",
    "vector_db": <faiss.Index>,           # FAISS向量索引对象
    "document": {
      "metainfo": { "company_name":"XX公司", ... },
      "content": { "chunks": [...], "pages": [...] }
    }
  },
  ...
]

🔍 retrieve_by_company_name() — 核心检索方法

①
定位目标报告——在 self.all_dbs 中按 company_name 匹配
②
生成 query embedding——调用 OpenAI text-embedding-3-large 模型
③
FAISS 向量检索——vector_db.search(embedding_array, k=top_n)，返回距离和索引
④
组装结果——根据 return_parent_pages 决定返回 chunk 还是整页
🐍 向量检索核心代码 
# 将query转为向量

    input=query, model="text-embedding-3-large"
).data[0].embedding

# FAISS搜索Top-K最相似的向量

📖 text-embedding-3-large 是什么？
  • OpenAI 最强的 embedding 模型，输出 3072 维向量
  • 支持多语言（中英文都行），最大输入 8191 token
  • 算出的"距离"越小 = 语义越相似（这里是余弦距离，用 FAISS 的 METRIC_INNER_PRODUCT 近似）

📄 retrieve_all() — 全量返回（兜底策略）

不进行任何检索，直接返回目标公司的所有页面，每页标记 distance=0.5。这是一个兜底策略—— 当检索结果都不好的时候，把全部内容给 LLM 看，让它自己找答案。

📏 get_strings_cosine_similarity() — 静态工具方法

计算两段文本的余弦相似度。每次调用都会重新创建 OpenAI 客户端和生成 embedding，适合独立调用 而非批量场景。返回 0~1 的值，越接近 1 越相似。

💡 return_parent_pages 的妙用：检索默认返回 chunk（~300 token 的小块），但有时候用户需要看整页上下文。 开启此选项后，即使只匹配到某个 chunk，也会返回该 chunk 所在的整页内容，提供更完整的上下文给下游 LLM。

③ HybridRetriever — 混合检索 + LLM 重排序 ADVANCED

🤔 这个类是做什么的？（大白话）

这是项目中真正用于问答的检索器。它分两步：① 先用向量检索拉回一堆候选（比如28个）， ② 再用 GPT-4o-mini 对这28个候选项逐一打分，"这个 chunk 和问题相关吗？有多相关？" 最后按加权组合分数（向量相似度 × 0.3 + LLM 评分 × 0.7）排序，只保留前6个。

🔧 参数说明

🤖 LLM Reranking 的两种模式

① 单文档模式（batch_size=1）
  • 每个文档独立发送给 LLM："这个文档和问题相关吗？打几分？"
  • 返回 relevance_score（0~1）+ reasoning（判断理由）
  • 优点：精确，每个文档得到独立评估
  • 缺点：慢（28个文档 = 28次 API 调用），但用了多线程并行加速
② 批量模式（batch_size>1，默认为2）
  • 每批2个文档一起发给 LLM："比较这2个文档，分别打分并排序"
  • 返回 block_rankings 数组，每个包含 relevance_score + reasoning
  • 优点：快（14批 vs 28次调用），LLM 能横向比较
  • 缺点：batch 之间的文档没有互相比较（这是两阶段检索的通用取舍）

🔢 加权融合公式

📐 组合分数计算 
combined_score = llm_weight × relevance_score + (1-llm_weight) × distance

# 默认：combined_score = 0.7 × LLM评分 + 0.3 × 向量相似度
# 最终按 combined_score 降序排列，取前 top_n 个
📖 注意：distance 和 similarity 的关系

向量检索返回的 distance 是余弦距离（越小越相似），但 LLM 返回的 relevance_score 是相关度（越大越相关）。两者方向相反！ 在计算 combined_score 时直接将它们加权相加，这意味着代码假设 FAISS 的距离值已经被处理为"越大越好"的方向 （例如通过 1-distance 转换，或者 FAISS 索引使用了内积而非距离）。

⚠️ 潜在问题：如果 FAISS 返回的是 L2 距离（越小越好），而 LLM relevance_score 是越大越好， 那么 combined_score = 0.7 × rel + 0.3 × dist 中 dist 越小反而让总分变小， 这会导致排序结果混乱。建议确认 FAISS 索引的实际度量方式（内积/余弦/L2）。
