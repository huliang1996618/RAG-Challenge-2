Q1：为什么要把表格"序列化"成文本块？直接用 Markdown 表格进向量库不行吗？

这个模块的核心任务是生成"上下文独立的文本块"。什么情况下 Markdown 表格在向量库中检索不到？"上下文独立"到底是什么意思？如果表格里只有一行 | 100亿 | 15亿 |，没有表头，LLM 能理解这是什么吗？

💡 提示：回顾 text_splitter.py 的切块机制——Markdown 表格被切成 300 token 的小块后，行列关系可能被切在两个不同 chunk 中。
📝 点击查看答案
Markdown 表格直接进向量库有两个致命问题：

① 被切分破坏：text_splitter 的 RecursiveCharacterTextSplitter 不认识表格边界。一个3列×30行的利润表（≈350 token）会被切成两个 chunk——chunk1 包含前20行，chunk2 包含后10行。表头只在 chunk1 中，chunk2 变成无头数据

② 上下文缺失：检索到 | 100亿 | 15亿 | 这样的行，LLM 不知道这是"营业收入100亿元同比增长15%"。序列化后的 information_block 是"营业收入在2024年为100亿元"——完整、自包含的句子

"上下文独立"的含义：每个 block 被检索到后，不需要依赖同一表格的其他行或表头就能被 LLM 正确理解。RAG 检索是"单片段命中"——你永远不知道用户会搜到哪个 block。

Q2：为什么给 LLM 的是 HTML 而不是 Markdown？

代码中 table_content = table_info["html"] 明确选择了 HTML。回顾前面的 Q4（三种格式问题），HTML 相比 Markdown 有哪些不可替代的优势？合并单元格（colspan="3"）用 Markdown 能表达吗？

💡 提示：年报财务报表有大量合并单元格（如"营业总成本"横跨3列），Markdown 会把它们"摊平"，LLM 会误判数据关系。
📝 点击查看答案
HTML 保留了 Markdown 无法表达的表格结构信息：

年报利润表中：<td colspan="3">营业总成本</td> → 表示"营业总成本"横跨3个子列（营业成本+税金+费用）
Markdown 输出：| 营业总成本 | 营业总成本 | 营业总成本 | → 重复3次，LLM 误以为有3个独立的"营业总成本"

LLM（GPT-4o）训练数据中包含大量 HTML，它能理解 colspan 和 rowspan 的语义含义。给 LLM 看原始的 HTML 表格 → 它理解的是和人类阅读 PDF 时完全一致的表格结构。这是 Markdown 的扁平化表示永远做不到的。

Q3：system_prompt 中 "context-independent" 是整个模块的灵魂，为什么？

"These blocks must be totally context-independent because they will be used as separate chunk to populate database."

如果一个信息块是"净利润：15亿元"，但没有说明是哪家公司、哪一年、什么币种，它被 RAG 检索到后 LLM 能正确回答吗？每个 block 需要包含什么才算"自给自足"？

💡 提示：RAG 是"片段匹配"——检索命中的只是其中一个 block，其他 block 不会被召回，所以每个 block 必须自足。
📝 点击查看答案
"context-independent" 解决了 RAG 检索的根本矛盾——片段 vs 完整上下文：

用户问"比亚迪2024年净利润多少？"→ 向量检索返回 block #47 → 内容是"15亿元"
如果 block 中没有"比亚迪"、"2024年"、"净利润"这些上下文词 → LLM 不知道这15亿是什么

正确的 context-independent block："比亚迪股份有限公司2024年度合并利润表中净利润为15亿元人民币"
包含：公司名 + 年份 + 指标名 + 数值 + 单位 → 自给自足的完整陈述

这就是 system_prompt 要求 LLM 输出的格式——每个 information_block 都是一个独立的小段落。
⚡ 并行机制问题

Q4：为什么有两层并行？ThreadPoolExecutor（文件级）+ asyncio（表格级）

外层用多线程处理多个 JSON 文件，内层用 asyncio 并发处理单个文件中的多个表格。为什么不只用一种？如果一个文件有 50 个表格，只用 ThreadPoolExecutor 会怎样？只用 asyncio 又会怎样？

💡 提示：API 调用是 IO 密集型（asyncio 擅长），文件读写也是 IO 操作（ThreadPool 擅长）。两者分工不同。process_directory_parallel + async_serialize_tables
📝 点击查看答案
两层并行解决了不同粒度的 IO 等待问题：

外层 ThreadPoolExecutor（文件级）：同时处理多个 JSON 文件。文件 I/O（读JSON+保存JSON）会阻塞线程，用多线程让 CPU 在文件等待间隙切换到其他文件

内层 asyncio（表格级）：单个文件内 30 个表格 → 同时发送 30 个 LLM API 请求。LLM 调用是网络 IO（等待500ms-2s），asyncio 在此期间不阻塞线程

只用 ThreadPoolExecutor：50个表格顺序调用 LLM → 50×2s=100s
只用 asyncio：无法同时处理多个文件 → N个文件串行
两层结合：文件并行 × 表格并发 → 最优

Q5：async_serialize_tables（异步）和 serialize_tables（同步）的核心区别？

异步版先把所有表格请求收集起来，一次性批量发送；同步版逐个发请求、逐个等回复。对一个有 30 个表格的报告，两种方式的总耗时大约差多少倍？为什么 pipeline 默认用异步版？

💡 提示：同步版 = 30 × 单次延迟；异步版 ≈ max(所有请求延迟)。考虑网络 RTT 通常在 500ms~2s 之间。
📝 点击查看答案
同步 vs 异步的时间差异是数量级的：

同步版（serialize_tables）：for table in tables: result = await llm_call(table) → 30个表格 × 2秒/个 = 60秒
异步版（async_serialize_tables）：tasks = [llm_call(t) for t in tables]; results = await asyncio.gather(*tasks) → 30个请求同时发出 → ≈2秒（最慢的那个）

加速比≈30倍。这就是为什么默认用异步版——对于单个报告30+表格、数百个报告的批量处理，同步版完全不可用。

Q6：_get_table_context() 为什么要提取表格前后的上下文？不提取会怎样？

代码提取了表格前的文本块和表格后的文本块作为上下文发给 LLM。举个例子：表格标题"合并利润表"在表格上方的段落中，如果不提取，LLM 只看到一张无名表格，能正确序列化吗？

💡 提示：年报中表格标题常常在表格上方的段落中，比如"下表列示了本公司 2024 年度..."。还有脚注解释在表格下方。
📝 点击查看答案
表格标题和脚注通常在 HTML 表格之外，不提取上下文 LLM 只能盲猜：

PDF 排版中，表格标题是独立段落（如"合并利润表"），表格主体是另一个元素。Docling 解析后它们分属不同的 children 节点。

不提取的后果：LLM 看到一张 3列×20行 的 HTML 表格，但不知道这是"利润表"还是"资产负债表" → 生成的 information_block 缺少表格标题 → "营业收入100亿"变成了"某数字100亿"

提取前后各 500 字符的上下文 → LLM 能看到"下表列示了本公司2024年度合并利润表" → 生成的 block 带上正确的表格名称和年份

Q7：Pydantic Schema 中 subject_core_entity 字段的设计有什么讲究？

每个 information_block 都有 subject_core_entity（如"营业收入"）。为什么需要这个字段？如果 chunk text 是"2024年为100亿元，2023年为80亿元"但没有主语，向量检索能匹配到"XX公司营收多少"吗？

💡 提示：向量检索依赖语义——"营收"的 embedding 和"2024年为100亿元"的 embedding，哪个更能匹配用户问题？subject 让每个 block 有了"标题"。
📝 点击查看答案
subject_core_entity 是每个 block 的"嵌入锚点"：

information_block text："2024年为100亿元，2023年为80亿元，同比增长25%"
这段 text 不含"营业收入"这个词 → embedding 和用户 query "营业收入多少"的相似度低

但 subject_core_entity="营业收入" 被拼入最终 chunk text：
"营业收入：2024年为100亿元，2023年为80亿元，同比增长25%"
→ embedding 包含"营业收入"语义 → 与用户 query 完美匹配

这就是 subject 的设计价值：它让每个数值 block 有了明确的"主题标签"，大幅提升向量检索的召回率。

Q8：serialized 字段被写入原 JSON 的位置有什么讲究？

看 async_serialize_tables() 中，serialized 被插入到紧挨着 html 的位置。下游 text_splitter.py 的 _get_serialized_tables_by_page() 怎么找到并使用它？如果写到了错误的位置会怎样？

💡 提示：回顾 text_splitter.py 的 _get_serialized_tables_by_page()——它检查 if 'serialized' not in table: continue，然后读取 table["serialized"]["information_blocks"]。
📝 点击查看答案
serialized 写入原 JSON 实现了"原地增强"而非"分离输出"：

原始 table 对象：{page:5, table_id:3, html:"...", md:"...", json:[...]}
写入后：{page:5, table_id:3, html:"...", md:"...", json:[...], serialized:{information_blocks:[...]}}

下游 _get_serialized_tables_by_page() 遍历所有 tables → 跳过无 serialized 字段的 → 读取 table["serialized"]["information_blocks"]

写错位置的后果：① 写到 table 对象外部 → 下游找不到 ② 字段改名 → if 'serialized' not in table 跳过

Q9：这个模块在整个 RAG 管线中是必选还是可选？不运行会怎样？

看 pipeline.py 的调用链和 TextSplitter 的构造参数：serialized_tables_dir 是 Optional。不运行 tables_serialization.py，表格数据还能被检索到吗？两种路径的检索效果差异有多大？

💡 提示：路径 A（Markdown 嵌入页文本）表格依然能检索，但可能被切散；路径 B（序列化独立块）每个数据行都是完整检索单元。这是渐进增强设计。
📝 点击查看答案
此模块是可选组件，属于"渐进增强"设计：

路径 A（不运行此模块）：表格以 Markdown 形式嵌入 page.text → 被 text_splitter 切分为 chunk → 存入向量库。效果：表格能被检索，但可能被切散（跨chunk），且缺少上下文独立的自然语言描述

路径 B（运行此模块）：额外生成 serialized_table chunk → 每个数据行都是独立检索单元 → "营业收入2024年100亿元"作为完整句子可被精确匹配

检索效果差异：用户问"比亚迪营收多少" → 路径A返回包含表格片段的 chunk（可能有无关数据）→ 路径B直接返回"营业收入：2024年100亿元"（精准命中）

Q10：一个大表格生成 100+ 个 information_block，对下游检索有什么影响？

上市公司完整利润表可能有 50-100 行，每行生成一个独立 block 存入向量库。好处是每行数据都能被精确检索到；坏处是什么？对 Top-K 检索有什么影响？

💡 提示：考虑向量库中 "serialized_table" 类型 chunk 的占比。如果 100 个表格 block 挤占了正文内容在 Top-K 中的位置，会不会导致 LLM 看不到用户真正需要的上下文？
📝 点击查看答案
100+ blocks 带来高召回的同时引入了"表格噪声污染"：

好处：每一行财务数据都是独立检索单元 → 搜索"营业成本"精确命中对应 block → 召回率极高

坏处（Top-K 污染）：200 chunk（100文本+100表格）。query "比亚迪2024年发展战略" → embedding 可能给表格 block 偏高的分数（数字和年份与正文语义重叠） → Top-28 中 20 个是表格 block → 正文"战略描述"被挤出 → LLM 只能从数字中猜战略 → 答案质量下降

缓解方案：① 检索时按 type 加权（content 优先于 serialized_table）② 限制每类在 Top-K 中的比例 ③ hybrid search 增加多样性

① TqdmLoggingHandler + process_messages() — 模块级工具 UTILITY

🤔 这些做什么？（大白话）

问题：在多线程并行处理时，tqdm 进度条和 logging 日志会互相"打架"——日志输出会破坏进度条的显示。 解决方案：用一个全局消息队列 message_queue 把日志暂存起来，等 tqdm 空闲时再统一输出。

🐍 Python — 关键代码 
message_queue = Queue()  # ═══ 全局消息队列 ═══

class TqdmLoggingHandler(logging.Handler):
    def emit(self, record):
        # ═══ 不直接输出，而是放入队列 ═══
        msg = self.format(record)
        message_queue.put((record.levelno, msg))

def process_messages():
    while not message_queue.empty():
        level, msg = message_queue.get_nowait()
        tqdm.write(msg)  # ═══ 用 tqdm.write 而非 print ═══
📖 图解说明
  • 为什么用 Queue？多个线程可能同时产生日志，Queue 是线程安全的，保证消息不丢失、不混乱
  • tqdm.write() 是 tqdm 提供的安全输出方法——它在输出前先清除当前进度行，输出后再重新绘制进度条
  • process_messages() 在主循环中定期被调用（见⑨），一次性清空队列中所有积压消息
📍 它在流程中的位置： TqdmLoggingHandler 在② __init__ 中被注册到 logger； process_messages() 在⑨ process_directory_parallel 的主循环中定期调用

② TableSerializer.__init__() — 初始化解析器 CONSTRUCTOR

🤔 这个函数做什么？（大白话）

继承父类 BaseOpenaiProcessor（提供 OpenAI API 调用能力）， 同时搭建自己的日志系统和临时文件目录。

🐍 Python — 关键代码 
class TableSerializer(BaseOpenaiProcessor):
    def __init__(self, preserve_temp_files: bool = True):
        super().__init__()  # ═══ 继承 OpenAI 客户端 ═══
        self.preserve_temp_files = preserve_temp_files
        os.makedirs('./temp', exist_ok=True)

        # ═══ 定制日志：用队列 + tqdm 兼容输出 ═══
        self.logger = logging.getLogger('TableSerializer')
        self.logger.handlers.clear()
        handler = TqdmLoggingHandler()
        handler.setFormatter(logging.Formatter('%(levelname)s: %(message)s'))
        self.logger.addHandler(handler)
        self.logger.propagate = False
📖 图解说明
  • handlers.clear()：清除默认 handler，避免日志重复输出
  • propagate = False：阻止日志向上传递到 root logger，确保只通过我们的 TqdmLoggingHandler 输出
  • preserve_temp_files：控制是否保留中间请求文件（调试用），默认为 True
📍 它在流程中的位置： 创建 TableSerializer 实例的第一步，为后续 API 调用和日志输出做准备

③ _get_table_context() — 提取表格上下文 CORE

🤔 这个函数做什么？（大白话）

在 PDF 解析产生的 JSON 报告中，找到目标表格所在的页面，然后截取它"上面最近的文字"和"下面最近的文字"， 作为 LLM 理解表格的辅助上下文。

比如：表格上方可能写着"下表展示了2023年各季度营收"，下方可能有脚注"单位：百万元"——这些文字对表格理解至关重要。

🐍 Python — 关键代码 
def _get_table_context(self, json_report, target_table_index):
    # ═══ 1. 找到目标表格的页码 ═══
    table_info = next(t for t in json_report["tables"] if t["table_id"] == target_table_index)
    page_num = table_info["page"]

    # ═══ 2. 获取该页面的内容块列表 ═══
    page_content = next(p["content"] for p in json_report["content"] if p["page"] == page_num)

    # ═══ 3. 定位目标表格在页面中的位置 ═══
    cur_pos = -1
    for i, block in enumerate(page_content):
        if block["type"] == "table" and block.get("table_id") == target_table_index:
            cur_pos = i; break

    # ═══ 4. 找上方最近的表格（作为上下文边界） ═══
    prev_table_pos = -1
    for i in range(cur_pos - 1, -1, -1):
        if page_content[i]["type"] == "table":
            prev_table_pos = i; break

    # ═══ 5. 截取上下文：上方从上一张表之后开始 ═══
    start = prev_table_pos + 1 if prev_table_pos != -1 else 0
    context_before = page_content[start:cur_pos]

    # ═══ 6. 截取下方最多3个文字块 ═══
    context_after = page_content[cur_pos + 1:cur_pos + 4]  # 最多3块

    # ═══ 7. 提取纯文本 ═══
    return "\n".join(b.get("text", "") for b in context_before), \
           "\n".join(b.get("text", "") for b in context_after)
📖 图解说明
  • 为什么需要找"上一张表"的位置？如果页面上有3张表，第2张表的上下文应该是第1张表之后、第2张表之前的文字——不能跨表取上下文，否则会张冠李戴
  • 下方最多取3个块：控制 token 消耗——实际代码中如果上下两张表之间有超过3个文字块，只取最近的3个
  • 空安全：如果目标表在页面上第一个位置（上方无文字），start=0，context_before 为空列表
📍 它在流程中的位置： 被⑤ _serialize_table 调用 → 返回的 context_before 和 context_after 传给④ _send_serialization_request 组装 prompt

④ _send_serialization_request() — 发请求给 LLM LLM CALL

🤔 这个函数做什么？（大白话）

把 HTML 表格 + 上下文文字拼成一段 prompt，发给 GPT-4o-mini， 并要求 LLM 按照 TableBlocksCollection 的 Pydantic schema 输出结构化 JSON。

🐍 Python — 关键代码 
def _send_serialization_request(self, table, context_before, context_after):
    user_prompt = ""

    # ═══ 拼装 user prompt ═══
    if context_before:
        user_prompt += f'Here is additional text before the table...\n"""{context_before}"""\n\n'
    user_prompt += f'Here is a table in HTML format:\n"""{table}"""'
    if context_after:
        user_prompt += f'\n\nHere is additional text after the table...\n"""{context_after}"""'

    # ═══ 调用父类方法：固定 temperature=0 ═══
    answer_dict = self.send_message(
        model='gpt-4o-mini-2024-07-18',
        temperature=0,                    # ═══ 确保输出稳定 ═══
        system_content=TableSerialization.system_prompt,
        human_content=user_prompt,
        is_structured=True,
        response_format=TableSerialization.TableBlocksCollection  # ═══ 约束输出结构 ═══
    )
    return answer_dict
📖 图解说明
  • temperature=0：要求 LLM 输出高度确定性——表格序列化不需要创意，需要准确
  • is_structured=True：告诉父类使用结构化输出模式（function calling / response_format），确保返回的是符合 Pydantic schema 的 JSON
  • 上下文是可选的：如果 context_before 为空（表格在页面顶部），就直接传入表格；同样，context_after 也可为空
  • count_tokens()：代码中还统计了输入和输出 token 数，用于成本追踪和调试
📍 它在流程中的位置： 被⑤ _serialize_table 调用 → 返回 Pydantic 对象（符合 TableBlocksCollection schema）

⑤ _serialize_table() — 单表序列化调度器 ORCHESTRATOR

🤔 这个函数做什么？（大白话）

它是单个表格序列化的"总指挥"：先调用③提取上下文，再取出表格的 HTML， 最后交给④发给 LLM——三步一气呵成。

🐍 Python — 关键代码 
def _serialize_table(self, json_report: dict, target_table_index: int) -> dict:
    # ═══ Step 1: 提取上下文 ═══
    context_before, context_after = self._get_table_context(json_report, target_table_index)

    # ═══ Step 2: 取出 HTML 表格 ═══
    table_info = next(t for t in json_report["tables"] if t["table_id"] == target_table_index)
    table_content = table_info["html"]

    # ═══ Step 3: 发请求 ═══
    result = self._send_serialization_request(
        table=table_content,
        context_before=context_before,
        context_after=context_after
    )
    return result
📖 图解说明
  • 为什么拆成三个函数？③负责数据定位，④负责 API 通信，⑤负责串联——各司其职，方便单元测试和复用
  • 两次查找 table_info：③中查一次（为了获取页码），⑤中再查一次（为了获取 HTML 内容）。这是为了让③和⑤各自独立，不互相依赖内部状态
  • 结果格式：返回的是 TableBlocksCollection Pydantic 对象，包含 subject_core_entities_list、relevant_headers_list、information_blocks
📍 它在流程中的位置： 被⑥ serialize_tables 和⑦ async_serialize_tables 调用 → 调用③和④完成实际工作

⑥ serialize_tables() — 同步处理所有表格 SYNC

🤔 这个函数做什么？（大白话）

遍历 JSON 报告中的每一张表格，逐个调用⑤进行序列化， 结果直接写入原 JSON 的 "serialized" 字段中。这是最简单的"一个接一个"的处理方式。

🐍 Python — 关键代码 
def serialize_tables(self, json_report: dict) -> dict:
    for table in json_report["tables"]:
        table_index = table["table_id"]

        # ═══ 对每张表：提取上下文 → 发 LLM 请求 → 获取结果 ═══
        serialization_result = self._serialize_table(json_report, table_index)

        # ═══ 结果直接写入原字典 ═══
        table["serialized"] = serialization_result

    return json_report
📖 图解说明
  • 原地修改（in-place）：直接在原 json_report 字典上添加 "serialized" 字段，不创建新数据结构
  • 串行 = 慢但简单：每张表都要等上一张处理完。如果一个文档有20张表，就是20次串行 API 调用，适合小批量处理
  • 实际使用：目前代码中主要通过⑦异步版本处理，⑥作为备选方案保留
📍 它在流程中的位置： 直接可被外部调用；在⑧ process_file 中通过⑦异步版本间接使用。目前已不直接使用⑥。

⑦ async_serialize_tables() — 异步并发处理 ASYNC

🤔 这个函数做什么？（大白话）

和⑥做同样的事，但用异步方式一次性发送所有表格的请求——对所有表格同时提取上下文、同时发送 LLM 请求， 等所有结果回来后再统一写回 JSON。速度远快于串行。

🐍 Python — 关键代码 
async def async_serialize_tables(self, json_report, ...):
    queries = []
    table_indices = []

    # ═══ 1. 批量准备所有请求数据 ═══
    for table in json_report["tables"]:
        table_index = table["table_id"]
        context_before, context_after = self._get_table_context(...)
        query = "..."  # 拼装 prompt
        queries.append(query)
        table_indices.append(table_index)

    # ═══ 2. 一次性提交所有请求 ═══
    results = await AsyncOpenaiProcessor().process_structured_ouputs_requests(
        model='gpt-4o-mini-2024-07-18',
        temperature=0,
        system_content=TableSerialization.system_prompt,
        queries=queries,
        response_format=TableSerialization.TableBlocksCollection,
    )

    # ═══ 3. 结果按顺序写回 ═══
    for table_index, result in zip(table_indices, results):
        # 在 html 字段后插入 serialized 字段
        new_table = {}
        for key, value in table_info.items():
            new_table[key] = value
            if key == "html":
                new_table["serialized"] = result["answer"]
        # 替换原数组中的元素
        json_report["tables"][i] = new_table

    return json_report
📖 图解说明
  • AsyncOpenaiProcessor：父类中定义的异步请求处理器，内部使用 aiohttp 并发发送 API 请求——类似"10个窗口同时办业务"
  • 插入位置策略：结果被插入到 "html" 字段之后——这样 JSON 序列化后的字段顺序是 table_id → ... → html → serialized → ...，保持可读性
  • zip() 保证顺序一致：table_indices 和 results 按相同顺序排列，确保结果写回正确的表格
📍 它在流程中的位置： 被⑧ process_file 调用 → 通过 AsyncOpenaiProcessor 并发调用 LLM API

⑧ process_file() — 处理单个 JSON 文件 FILE HANDLER

🤔 这个函数做什么？（大白话）

读取一个 JSON 文件 → 调用⑦异步序列化所有表格 → 写回同一个 JSON 文件。 因为⑦是 async，所以这里需要创建事件循环来运行它。

🐍 Python — 关键代码 
def process_file(self, json_path: Path) -> None:
    with open(json_path, 'r', encoding='utf-8') as f:
        json_report = json.load(f)

    thread_id = threading.get_ident()  # ═══ 每个线程用独立的临时文件 ═══

    # ═══ 创建事件循环运行异步方法 ═══
    loop = asyncio.new_event_loop()
    asyncio.set_event_loop(loop)
    updated_report = loop.run_until_complete(
        self.async_serialize_tables(json_report, ...)
    )
    loop.close()

    # ═══ 写回 ═══
    with open(json_path, 'w', encoding='utf-8') as f:
        json.dump(updated_report, f, indent=2, ensure_ascii=False)
📖 图解说明
  • 为什么创建新事件循环？因为⑨在多线程中调用 process_file，每个线程需要独立的事件循环
  • thread_id 用于临时文件名：避免多线程同时写同一个临时文件导致冲突
  • 临时文件清理：请求文件和结果文件在使用后通过 os.remove() 删除（除非 preserve_temp_files=True）
📍 它在流程中的位置： 被⑨ process_directory_parallel 通过线程池调用 → 内部调用⑦完成实际序列化

⑨ process_directory_parallel() — 并行处理整个目录 PARALLEL

🤔 这个函数做什么？（大白话）

扫描目录下所有 JSON 文件，用线程池并行处理——每个线程负责一个文件， 线程内部用异步方式处理该文件的所有表格。这是两重并行的叠加。

🐍 Python — 关键代码 
def process_directory_parallel(self, input_dir: Path, max_workers: int = 5):
    json_files = list(input_dir.glob("*.json"))

    with ThreadPoolExecutor(max_workers=max_workers) as executor:
        with tqdm(total=len(json_files), desc="Processing files") as pbar:
            futures = []
            for json_file in json_files:
                future = executor.submit(self.process_file, json_file)
                future.add_done_callback(lambda p: pbar.update(1))
                futures.append(future)

            # ═══ 等待所有线程完成 ═══
            while futures:
                process_messages()  # ═══ 定期输出积压的日志 ═══
                # 检查哪些 future 已完成
                for future in list(futures):
                    if future.done():
                        futures.remove(future)
                        try: future.result()
                        except Exception: self.logger.error(...)
                time.sleep(0.1)
📖 图解说明
  • 两层并行：
    • 外层（线程池）：5个线程同时处理5个不同的 JSON 文件
    • 内层（异步）：每个线程内部对文件中的多张表异步并发请求 LLM
    比如 3 个文件各含 4 张表 = 最多 5×4 = 20 个并发 API 请求
  • process_messages() 轮询：每 0.1 秒检查一次消息队列，确保不丢失日志输出
  • add_done_callback：每当一个 future 完成，自动更新 tqdm 进度条，不需要手动等待
📍 它在流程中的位置： 最终入口函数，是整个模块的"启动按钮" → 提交任务到线程池 → 每个线程执行⑧

⑩ TableSerialization() — Prompt 与 Pydantic 模型 SCHEMA

🤔 这个做什么？（大白话）

它是纯静态配置类，定义了 LLM 的系统提示词（告诉 LLM 它要做什么） 和输出结构（规定 LLM 必须返回什么格式的 JSON）。这是整个模块的"规则书"。

📋 System Prompt — LLM 的"工作说明书"

📝 System Prompt 
system_prompt = (
    "You are a table serialization agent.\n"
    "Your task is to create a set of contextually independent blocks"
    " of information based on the provided table and surrounding text.\n"
    "These blocks must be totally context-independent because they will"
    " be used as separate chunk to populate database."
)

核心指令："创建上下文独立的文本块"——这正是 RAG 场景的需求： 每个块单独被检索到时，读者不需要看原始 PDF 就能理解其含义。

📐 Pydantic 模型结构

🐍 数据结构定义 
class SerializedInformationBlock(BaseModel):
    subject_core_entity: str      # ═══ 主体实体（通常是行头） ═══
    information_block: str       # ═══ 完整上下文信息块 ═══
        # 包含：1.所有相关表头  2.单位及描述  3.表格名称/脚注/货币等

class TableBlocksCollection(BaseModel):
    subject_core_entities_list: List[str]    # ═══ 所有主体实体列表 ═══
    relevant_headers_list: List[str]         # ═══ 所有相关表头 ═══
    information_blocks: List[SerializedInformationBlock]  # ═══ 信息块列表 ═══
📖 图解说明
  • SerializedInformationBlock：一个"烘焙好的"文本单元。LLM 需要从表格中取一个行头作为 subject_core_entity，然后把该行所有列的值 + 单位 + 脚注 etc. 融合成一段流畅文本
  • 相邻行可以合并：如果表格中某行的数据离开相邻行无法理解，LLM 可以把多行信息合并到一个 block 中（参见 Field description 中的说明）
  • "惩罚条款"："SKIPPING ANY VALUABLE INFORMATION WILL BE HEAVILY PENALIZED!" — 这行 caps 提示词确保 LLM 不会偷懒漏信息
  • TableBlocksCollection：顶层容器，汇总了所有行实体列表 + 表头列表 + 信息块列表，方便后续按实体名检索
📍 它在流程中的位置： 被④ _send_serialization_request 引用其 system_prompt 和 TableBlocksCollection schema； 被⑦ async_serialize_tables 同样引用。是整个序列化逻辑的"宪法"。
