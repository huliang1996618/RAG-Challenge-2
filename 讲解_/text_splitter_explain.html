🔥 核心架构

Q1：RecursiveCharacterTextSplitter 到底"递归"在哪里？为什么不用简单的固定长度切分？
如果直接用 text[:300]、text[300:600] 切分，会有什么问题？分隔符 ["\n\n", "\n", " ", ""] 的分层逻辑是什么——它不是在"找切分点"，而是在"找最不破坏语义的切分点"。
💡 提示：参看 ④ _split_page
📝 点击查看答案
RecursiveCharacterTextSplitter 的"递归"体现在分隔符优先级逐级降级：

它不是直接按300 token切分，而是按顺序尝试分隔符 ["\n\n", "\n", " ", ""]：
① 先用 \n\n（段落边界）切分 → 检查每段是否≤300 token
② 超长的段再用 \n（换行/句子边界）切分
③ 还超长就用 " "（单词边界）
④ 最后才按字符硬切

如果直接用 text[:300] 固定长度切分，会出现：单词被从中间劈开（"营业收|入"）、句子被截断（"公司2024年营收为|100亿元"），导致 embedding 语义严重失真。RecursiveSplitter 的核心理念是："在不得不切分时，选择对语义破坏最小的位置"。
Q2：chunk_size=300、chunk_overlap=50 是怎么定出来的？
如果改成 chunk_size=100 或 1000，下游检索会怎样变化？overlap=50 意味着相邻 chunk 有 16.7% 的重叠率——这个比例合理吗？为什么不设成 overlap=chunk_size/2？
💡 提示：_split_page() 的注释说 "The original text includes markdown tables."——和 chunk_size 选择有什么关系？
📝 点击查看答案
300 token 是一个经验最优值，在"精度"和"完整性"之间取得平衡：

• chunk_size=100：信息碎片化严重。一个完整的 Markdown 表格（约150-200 token）会被切成2-3块，行列关系破坏，检索时"营收100亿"可能和"2024年"分在两个chunk中
• chunk_size=1000：检索精度低。用户搜"营业成本"，返回的chunk包含一整页内容（利润表+资产负债表+文字说明），LLM被大量噪音淹没
• 300 token（≈150-200个中文字）：刚好能容纳一个典型表格或2-3个完整段落，同时足够小以保证检索精度

overlap=50（16.7%）：保证关键信息跨chunk边界的连续性。"公司2024年营收为"（chunk1末尾）+ "100亿元"（chunk2开头）→ 50 token重叠让两个chunk都包含完整句子的核心部分。
为什么不是 overlap=150（50%）？→ 50%的冗余过高，向量库存储膨胀3倍，检索时大量重复内容降低排序质量。

⚡ 设计决策

Q3：为什么 content 和 serialized_table 两种 chunk 共用一个递增 ID？
如果文本 chunk 和表格 chunk 各自独立编号（T-1, T-2 和 S-1, S-2），会丢失什么信息？共用 ID 对下游 retrieval.py 有没有帮助？ingestion.py 建向量库时在乎这个 ID 吗？
💡 提示：搜索时可能同时召回同一页的文本 chunk 和表格 chunk——ID 顺序能告诉你原始页面的阅读顺序。
📝 点击查看答案
共用递增ID的核心价值是保留原页面的"阅读顺序"：

某报告第3页：用户先看到一段文字描述"公司2024年营业总收入同比增长15%"，然后看到一个表格详细列出各项收入数据。
在chunk数组中 → id=7（text chunk），id=8（serialized_table chunk）

如果各自独立编号（T-1, T-2, ... 和 S-1, S-2, ...），排序后表格可能出现在文本前面，打破了原始文档的叙事逻辑。

对下游的影响：ingestion.py 将 id 存入 FAISS 元数据，retrieval.py 检索到多个chunk后可先按id排序还原顺序，再放入LLM prompt——LLM就能按"先描述再表格"的自然顺序理解信息。
Q4：split_all_reports()
对比 pdf_parsing.py（有 ProcessPoolExecutor）和 tables_serialization.py（有 async LLM 调用），text_splitter.py 为什么用最简单的 for 循环？如果增加到 10000 个报告，瓶颈在哪里——CPU、I/O、还是内存？
💡 提示：tiktoken 编码和 RecursiveCharacterTextSplitter 都是纯 CPU 操作。一个 108 行的模块需要并行框架吗？
📝 点击查看答案
text_splitter 是纯 CPU 密集型、无 IO 瓶颈的模块，并行反而可能更慢：

• 操作速度：单次 tiktoken 编码 ≈ 微秒级，split_text ≈ 毫秒级。10000个文件的总处理时间可能在数分钟内完成
• 并行开销 > 收益：ProcessPoolExecutor 的进程创建（数百MB内存复制）+ pickle 序列化成本，远超 split_text 的计算量
• 对比 pdf_parsing：Docling 需要加载CNN/OCR模型（秒级）+ GPU推理 → 并行加速明显；text_splitter 没有模型，只有字符串操作
• 对比 tables_serialization：LLM API调用是IO等待（网络往返≈500ms）→ asyncio 并行可显著节省时间；text_splitter 没有网络IO

真正的瓶颈是磁盘I/O（读JSON+写JSON），但Python GIL下多线程对I/O操作已足够高效。

📊 数据处理

Q5：_get_serialized_tables_by_page() 用 "\n".join() 拼接 information_blocks——这个拼接会丢失什么信息？
每个 information_block 原本是独立的键值对，用换行符拼接后变成一段连续文本。这段文本进向量库后，语义检索还能精确匹配到某个 information_block 吗？table_id 字段是存给谁看的？
💡 提示：参看表格 chunk 的四个字段——table_id、page、text、type——哪些在向量库中，哪些只在 JSON 中？
📝 点击查看答案
\n.join() 拼接会丢失两层信息：

① 结构信息丢失：每个 information_block 原本是独立的键值对（如"行：营业收入 | 列：2024年 | 值：100亿元"），拼接后变成连续段落。语义检索时，embedding 是对整段文本编码，无法精确定位到"营业收入"这个具体指标
② 行/列对应关系丢失：拼接只是顺序排列，原始的行列交叉关系（同一行不同列的多个值）变得模糊

table_id 的作用：它是"回查键"。检索命中这个 chunk 后，通过 table_id 可以回到原始 tables_serialization.py 的 JSON 输出中，获取完整的 HTML/Markdown/JSON 三格式表格。text 进向量库做检索，table_id 做精确回查。

四字段分工：进向量库的是 text（embedding编码）；page、table_id、type 作为 FAISS 元数据存储，不参与向量检索但用于结果过滤和排序。
Q6：_split_page() 生成的 chunk 只有 page、text、length_tokens
对比 _split_report() 补上的 id 和 type 字段——为什么这两个字段不在 _split_page() 内部生成？如果 _split_page() 也加上 id，会出现什么并发问题？这是一种什么设计思想？
💡 提示：有状态的函数 vs 无状态的函数——哪个更容易测试和复用？
📝 点击查看答案
这是"单一职责原则"的典型体现——纯函数 vs 编排函数的分离：

_split_page() 是纯函数（无状态）：
• 输入：一个 page 对象 → 输出：chunks 列表
• 不依赖外部状态（不知道 chunk_id 现在是几号）
• 可以被独立单元测试：mock 一个 page，验证切分结果
• 可以在不同上下文中复用（比如批量处理和单页调试）

_split_report() 是编排函数（有状态）：
• 维护 chunk_id 计数器，协调文本chunk和表格chunk的顺序
• 知道全局上下文（当前是第几页、之前有多少chunk）

如果 _split_page() 也生成 id，要传入 start_id 参数 → 变成有状态函数 → 测试时必须构造完整上下文 → 增加了耦合度。这是 关注点分离 的经典实践。
Q7：serialized_tables_path 找不到时会打印 Warning
代码中 if not serialized_tables_path.exists(): print("Warning: ...") 后没有 raise。一份报告可以在没有序列化表格的情况下完成切分——这对下游还算"完整"吗？
💡 提示：_split_report() 的注释提到 "preserving markdown tables in content"——表格有几种来源？
📝 点击查看答案
这是"优雅降级"策略——表格有两个独立来源，缺一个也不影响基本功能：

来源1 — Markdown表格（始终存在）：
pdf_parsing.py 的 assemble_tables() 已经将表格以 Markdown 格式嵌入到每页的 text 中。_split_page() 切分时这些 Markdown 表格会随文本一起被切成 chunk。这是表格数据在向量库中的基本表示。

来源2 — 序列化表格（可选附加）：
tables_serialization.py 将表格转为"上下文独立的自然语言描述"（如"营业收入在2024年为100亿元"），这些 description block 作为独立的 serialized_table chunk 追加。

为什么 Warning 而非 Exception：
如果抛出异常 → 1000份报告中1份的序列化表格文件丢失 → 整批处理中断。
Warning继续 → 该报告仍有 Markdown 表格的向量表示，检索不会完全失败，只是缺少了"上下文独立"的增强描述。

Q8：追踪一个 chunk 的完整旅程——从 _split_page() 到最终被 LLM 消费。
_split_page() → [哪个模块?] → [什么数据结构?] → 用户提问时如何被检索? → LLM prompt 里长什么样?
在这个过程中，length_tokens 字段被用在哪里？type 字段被用在哪里？
💡 提示：画一条从 text_splitter.py → ingestion.py → retrieval.py → LLM 的完整链。
📝 点击查看答案
一个 chunk 的完整生命周期（6步）：

① text_splitter._split_page() → 输出 {page:3, text:"公司2024年营收...", length_tokens:287}
↓
② text_splitter._split_report() → 补充 id:42, type:"content"，写入JSON文件
↓
③ ingestion.py → 调用 OpenAI Embedding API 将 text 转为 3072维向量 → 存入 FAISS IndexFlatIP，元数据(id, page, type, sha1_name)存入 self.metadata[42]
↓
④ retrieval.py VectorRetriever → 用户query也转为向量 → index.search(query_vec, k=28) → 返回 distances=[0.87, 0.82, ...] (内积相似度) 和 indices=[42, 15, 98, ...] (chunk id)
↓
⑤ retrieval.py HybridRetriever → 通过 indices 从 metadata 取出对应 text → 打包成 batch → LLMReranker.rerank_documents() 评分 → 0.7×LLM + 0.3×distance 融合 → 取 top 6
↓
⑥ LLM Prompt → 6个chunk的 text 拼接为 context → "请根据以下文档内容回答问题：\n[文档1]公司2024年营收...\n[文档2]..." → LLM 生成最终答案

length_tokens 的使用：metadata 字段，用于估算 prompt token 消耗（确保6个chunk不超出LLM context window）
type 字段的使用：retrieval 结果中区分 content chunk 和 serialized_table chunk，可能用于调整 LLM 的阅读策略
Q9：如果必须删掉一个函数，哪个破坏最小？哪个最大？
删掉 count_tokens()：下游谁会受影响？删掉 _get_serialized_tables_by_page()：表格已经在 Markdown 文本里了，够用吗？删掉 _split_report()：还能叫 text_splitter 吗？删掉 split_all_reports()：能用 5 行代码替代吗？
把这四个函数按"可替代性"从高到低排序，给出你的理由。
📝 点击查看答案
按"可替代性"从高到低排序（删掉影响从小到大）：

① split_all_reports() — 可替代性最高
只是一个 for 循环 + 文件读写，5行代码即可替代：
for path in dir.glob("*.json"): data=json.load(open(path)); ...
删掉影响：需要调用方自己写遍历逻辑，模块失去"一键批处理"便利性

② count_tokens() — 可替代性高
单行代码 len(tiktoken.get_encoding("o200k_base").encode(text)) 即可替代
删掉影响：每个 chunk 不再记录 length_tokens，下游无法做 token 预算估算

③ _get_serialized_tables_by_page() — 可替代性中等
表格的两个来源（Markdown + 序列化描述）中只丢失了增强描述部分
删掉影响：检索结果缺少"上下文独立的自然语言表格描述"，但 Markdown 表格仍在 text 中可被召回

④ _split_page() — 可替代性低
封装了 RecursiveCharacterTextSplitter 的配置和调用，删掉后需要每个调用方自己配置
删掉影响：_split_report 的核心切分逻辑消失，但可用内联替代

⑤ _split_report() — 不可替代（破坏最大）
这是整个模块的"大脑"：协调文本切分、表格混入、ID分配、type标记。删掉它 → 所有 chunk 失去 id/type → ingestion 无法建索引 → 整个 RAG 检索瘫痪

① count_tokens() — Token 计数器 UTILITY

🤔 这个函数做什么？（大白话）

给一段文字，告诉你有多少个 token（GPT 眼中的"字数"）。 它用的是 OpenAI 的 tiktoken 库，编码器是 o200k_base（GPT-4o 使用的编码）， 所以算出来的 token 数和调用 GPT-4o API 时计费的方式完全一致。

🐍 关键代码 
def count_tokens(self, string: str, encoding_name="o200k_base"):
    encoding = tiktoken.get_encoding(encoding_name)  # 获取GPT-4o的编码器
    tokens = encoding.encode(string)                  # 文本 → token编号列表
    token_count = len(tokens)                         # 统计数量
    return token_count
📖 为什么用 o200k_base 而不是 cl100k_base？
  • cl100k_base 是 GPT-4/GPT-3.5 的编码器
  • o200k_base 是 GPT-4o 的编码器，对中文等多语言更高效
  • 这个项目最终用 GPT-4o 做问答，所以用 o200k_base 才能保证 chunk 大小和实际 API 计费一致
📍 它在流程中的位置： 被 _split_page() 和 _get_serialized_tables_by_page() 调用。 每个 chunk 生成后都会调用它来记录 length_tokens 字段。

② split_all_reports() — 批量入口 ENTRY

🤔 这个函数做什么？（大白话）

这是外部调用的入口函数。它扫描一个目录下的所有 .json 文件， 逐个读取、切分、保存。如果提供了序列化表格目录，还会为每个报告找到对应的表格文件。

🐍 关键代码 — 三步循环 
def split_all_reports(self, all_report_dir: Path, output_dir: Path,
                      serialized_tables_dir: Optional[Path] = None):
    all_report_paths = list(all_report_dir.glob("*.json"))   # ① 收集所有JSON文件

    for report_path in all_report_paths:
        # ② 查找对应的序列化表格文件（可选）
        serialized_tables_path = None
        if serialized_tables_dir is not None:
            serialized_tables_path = serialized_tables_dir / report_path.name

        # ③ 读取 → 切分 → 保存
        with open(report_path, 'r', encoding='utf-8') as file:
            report_data = json.load(file)
        updated_report = self._split_report(report_data, serialized_tables_path)
        output_dir.mkdir(parents=True, exist_ok=True)
        with open(output_dir / report_path.name, 'w', encoding='utf-8') as file:
            json.dump(updated_report, file, indent=2, ensure_ascii=False)
📖 参数详解
  • all_report_dir：上一道工序（parsed_reports_merging.py）输出的 JSON 所在目录
  • output_dir：切分后的 JSON 输出到哪里（下游 ingestion.py 会从这里读）
  • serialized_tables_dir：可选。如果提供了 tables_serialization.py 的输出目录，就会把表格 chunks 混入文本 chunks 中
📍 它在流程中的位置： RAG 管线的第3步。上游输入来自 parsed_reports_merging.py，输出给 ingestion.py。

③ _split_report() — 核心切分逻辑 CORE

🤔 这个函数做什么？（大白话）

这是整个模块的灵魂函数。它对一份报告做两件事： ① 把每页文本切成小块（chunk）；② 如果那一页有序列化表格，把表格也作为独立 chunk 插入。 最后给所有 chunk 分配全局唯一递增 ID。

🐍 关键代码 — 双重循环 
def _split_report(self, file_content, serialized_tables_report_path=None):
    chunks = []
    chunk_id = 0

    # 预先加载序列化表格，按页码分组（可选）
    tables_by_page = {}
    if serialized_tables_report_path is not None:
        with open(serialized_tables_report_path, 'r') as f:
            parsed_report = json.load(f)
        tables_by_page = self._get_serialized_tables_by_page(
            parsed_report.get('tables', []))

    for page in file_content['content']['pages']:
        # ─── ① 对本页文本切分成 chunks ───
        page_chunks = self._split_page(page)
        for chunk in page_chunks:
            chunk['id'] = chunk_id
            chunk['type'] = 'content'        # 标记为"文本类型"
            chunk_id += 1
            chunks.append(chunk)

        # ─── ② 如果本页有序列化表格，追加为独立 chunks ───
        if tables_by_page and page['page'] in tables_by_page:
            for table in tables_by_page[page['page']]:
                table['id'] = chunk_id
                table['type'] = 'serialized_table'  # 标记为"表格类型"
                chunk_id += 1
                chunks.append(table)

    file_content['content']['chunks'] = chunks
    return file_content
📖 两种 chunk 类型
  • content（文本 chunk）：本页文本切出来的小块，包含 page、text、length_tokens
  • serialized_table（表格 chunk）：从 tables_serialization.py 来的，包含 page、text（表格的描述性文本）、table_id
  • 两种 chunk 共享同一个递增 ID，确保检索时能按顺序还原
💡 为什么要把表格单独作为 chunk？
年报中的财务报表包含大量数字（营收、利润等），这些数字往往是问题的答案。 如果把表格和正文混在一起切分，表格会被切断，导致检索时找不到完整数据。 单独作为 chunk 可以保证每个表格是一个完整的检索单元。
📍 它在流程中的位置： 被 split_all_reports() 调用。它的输出是填充了 chunks 数组的完整报告 JSON。

④ _split_page() — 单页文本切分 CORE

🤔 这个函数做什么？（大白话）

把一页的长文本切成多个小块（chunk），每个 chunk 约 300 token， 相邻 chunk 之间重叠 50 token。用的是 LangChain 的 RecursiveCharacterTextSplitter。

🐍 关键代码 
def _split_page(self, page, chunk_size=300, chunk_overlap=50):
    # 创建基于 GPT-4o tokenizer 的分隔器
    text_splitter = RecursiveCharacterTextSplitter.from_tiktoken_encoder(
        model_name="gpt-4o",        # 使用GPT-4o的tokenizer
        chunk_size=chunk_size,      # 每个chunk最多300 tokens
        chunk_overlap=chunk_overlap # 相邻chunk重叠50 tokens
    )
    chunks = text_splitter.split_text(page['text'])

    chunks_with_meta = []
    for chunk in chunks:
        chunks_with_meta.append({
            "page": page['page'],              # 保留页码
            "length_tokens": self.count_tokens(chunk),  # 记录token数
            "text": chunk                       # 切分后的文本
        })
    return chunks_with_meta
📖 RecursiveCharacterTextSplitter 的工作原理
  1. 它用一组分隔符优先级来切分：先按段落（\n\n），再按换行（\n），再按空格，最后按字符
  2. 每次切分后检查 chunk 是否超过 300 token，超了就继续用下一级分隔符再切
  3. 这样能尽量在语义边界处切分，而不是把一个词劈成两半
📖 为什么需要重叠（overlap）？

假设一句话是："公司2024年营收为100亿元，同比增长20%。"
不加重叠：chunk1 = "公司2024年营收为"，chunk2 = "100亿元，同比增长20%。"→ 用户搜"2024年营收"只能找到 chunk1，但答案"100亿元"在 chunk2，检索不到！
加了50 token重叠：chunk1 = "公司2024年营收为100亿"，chunk2 = "营收为100亿元，同比增长20%。"→ 两个chunk都包含关键信息，检索召回率更高。

📍 它在流程中的位置： 被 _split_report() 对每一页调用。是实际执行切分操作的"刀片"。

⑤ _get_serialized_tables_by_page() — 表格按页分组 UTILITY

🤔 这个函数做什么？（大白话）

从 tables_serialization.py 的输出中读取表格数据， 把每个表格的 information_blocks（描述表格内容的文本块）拼接成一段文字， 然后按页码分组。这样 _split_report() 就能快速知道"第3页有哪些表格需要追加"。

🐍 关键代码 
def _get_serialized_tables_by_page(self, tables):
    tables_by_page = {}
    for table in tables:
        if 'serialized' not in table:       # 跳过未序列化的表格
            continue

        page = table['page']
        if page not in tables_by_page:
            tables_by_page[page] = []

        # 核心：将 information_blocks 拼接为一段文本
        table_text = "\n".join(
            block["information_block"]
            for block in table["serialized"]["information_blocks"]
        )

        tables_by_page[page].append({
            "page": page,
            "text": table_text,                          # 拼接后的描述文本
            "table_id": table["table_id"],               # 表格编号
            "length_tokens": self.count_tokens(table_text)  # token数
        })

    return tables_by_page
    # 返回：{3: [{page:3, text:"...", table_id:5, ...}], 5: [...]}
📖 information_block 是什么？

在 tables_serialization.py 中，每个表格被"序列化"为一组 information_block， 比如：
"表格标题：2024年度合并利润表"
"行：营业收入 | 列：2024年 | 值：100亿元"
这些块拼起来就是对这个表格的自然语言描述，可以直接当作文本 chunk 存入向量库。

📍 它在流程中的位置： 被 _split_report() 调用（仅当提供了序列化表格路径时）。内部调用了 count_tokens()。

📊 关键参数总结

参数 值 含义
chunk_size 300 tokens 每个chunk最多300个GPT-4o token（约200-250个英文单词或150-200个中文字）
chunk_overlap 50 tokens 相邻chunk重叠50 token，防止关键信息被切断
tokenizer o200k_base (gpt-4o) 与下游LLM计费方式一致
splitter RecursiveCharacterTextSplitter 优先在段落/句子边界切分，保持语义完整
👶 新手总结：text_splitter.py 做的事情可以用一句话概括——
"把每页长文本切成300 token的小块，相邻块重叠50 token，如果某页有表格就插进去"。

它虽然只有108行代码，但在 RAG 管线中非常关键：切得太大→检索精度低；切得太小→语义丢失。 300 token 是一个经验值，在"精度"和"完整性"之间取得了平衡。
